# V0.4b Real CLI Experience ‚Äî Implementation Plan

**Spec:** [`docs/superpowers/specs/2026-05-29-v0.4b-cli-experience-design.md`](../specs/2026-05-29-v0.4b-cli-experience-design.md)

**Goal:** Make the `modi` CLI feel like a real tool ‚Äî live streaming output with `rich`, colored tool activity markers, interactive approval prompts, TTY auto-detection.

**Tech:** `rich>=13.7`, existing stack (`pytest-asyncio` for async tests). **Hard rule: never pin langgraph to an older line ‚Äî always ride latest.**

**Test discipline:** TDD per task. Each task = write failing test ‚Üí minimal impl ‚Üí green ‚Üí commit. Push at end of each milestone.

## N1 ‚Äî Approval Prompt

- [x] **N1.1** Create `src/modi_harness/cli/prompt.py` with `ApprovalPrompt` class. Constructor takes a `Console`. Method `ask(approval: dict, agent: AgentProfile | None = None) -> tuple[str, str | None]`:
  - Returns `(decision, reason)` where decision is `"approved"` or `"rejected"`, reason is None for approval, str for rejection
  - Prints approval panel with tool name, args, risk, decision_kind
  - Uses `rich.prompt.Prompt.ask("[a]pprove [r]eject [d]etails", choices=["a","r","d"])`

  - On `"r"`: prints `Reason: ` and reads a single line via `Prompt.ask("Reason")`
  
  TDD via `tests/cli/test_prompt.py` ‚Äî patch `rich.prompt.Prompt.ask` with a mock that returns scripted responses. Cases: approve, reject (with reason), details-then-approve, details-then-reject.

- [x] **N1.2** Full suite green. Commit.

**Exit gate:** `ApprovalPrompt.ask` returns the right decision/reason for all three keypress paths; details path loops correctly.

## N2 ‚Äî Streaming Runner

- [x] **N2.1** Create `src/modi_harness/cli/runner.py` with `async def run_streaming(harness, *, agent: str, input: dict, thread_id: str | None, permission_mode: str | None, console: Console) -> int`:
  - Construct `StreamRenderer(console)` and `ApprovalPrompt(console)`
  - Print header: `[<agent>] running...`
  - Track start time
  - `async for event in harness.astream(...)`:
    - If `event_type == "approval_request"`:
      - Capture `thread_id`, `approval_id` from payload
      - Call `prompt.ask(payload)` to get decision
      - Break out of stream loop (astream completes when paused)
      - Call `harness.approve_action(thread_id, approval_id)` or `harness.reject_action(thread_id, approval_id, reason)` ‚Üí returns `RunTaskResponse`
      - Render the resulting messages/output post-hoc by passing them to renderer as synthetic events (or just print final output via `console.print`)
      - Continue stream loop with a new `astream` that resumes? ‚Äî simpler: terminate after the resume_task call and treat its return as the final event
    - Else: pass to `renderer.render_event(event)`
  - Print final terminal status with elapsed time
  - Return 0 if status is `completed`, 1 otherwise
  
  TDD via `tests/cli/test_runner.py` ‚Äî set up a harness with a scripted model that:
  - Case 1: returns text immediately ‚Üí assert exit code 0, output contains text
  - Case 2: produces an approval_request, scripted prompt approves ‚Üí assert exit code 0
  - Case 3: produces an approval_request, scripted prompt rejects ‚Üí assert exit code != 0 or status == rejected

- [x] **N2.2** Full suite green. Commit.

**Exit gate:** `run_streaming` handles happy path, approval-approved, approval-rejected; integration test verifies full flow.

## Risks watch

- **`rich` interaction with stdout buffering** ‚Äî async streaming may double-buffer; force `console.print(..., end="", flush=True)` for `model_delta`.
- **Single-keypress on Windows / non-Unix terminals** ‚Äî `rich.prompt.Prompt` falls back to enter-required input on terminals that don't support raw mode. Acceptable for V0.4b.
- **astream resumption after approval** ‚Äî current design renders post-approval output post-hoc rather than streaming. If user wants token-level streaming after approval, revisit in V0.5.
- **TTY detection in CI** ‚Äî CI runs are non-TTY by default; default to JSON in CI so test scripts work unchanged.
