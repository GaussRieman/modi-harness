  # V0.6.b Design Spec — Core Concepts Alignment

## 1. Purpose

V0.6.b aligns Modi Harness vocabulary around first principles. It does not add
runtime features. It defines the minimum stable concepts developers need in
order to reason about execution, persistence, context, memory, and trace.

The release corrects two sources of confusion:

- `workspace` was used as both a business work boundary and a per-run file
  directory.
- `memory` was used for both model-facing reusable knowledge and runtime-selected
  context records.

## 2. Non-Goals

- Do not add new runtime managers.
- Do not rename public APIs in this release.
- Do not change storage layout in this release.
- Do not modify examples as the main deliverable.
- Do not make Draft, Artifact, Log, or Project first-class concepts.
- Do not introduce a Memory governance product surface.

## 3. Core Concepts

V0.6.b recognizes eight terms. Four are primary user-facing concepts:

- `Workspace`
- `Context`
- `Memory`
- `Trace`

Four are execution and implementation concepts:

- `Session`
- `Thread`
- `Run`
- `Store`

### Workspace

Workspace is the work boundary.

It answers: where does this related work belong?

It may be a code repository, research topic directory, customer case, data
analysis package, or any other application-defined work boundary. It is not
limited to coding.

Workspace should be explicit. If no workspace is provided, Harness may create a
temporary workspace under the current directory:

```text
./.modi/workspaces/<workspace_id>/
```

Temporary workspaces are isolated Harness storage. They should be visible in
trace/output and should not enable workspace-scoped memory by default.

### Session

Session is the runtime container.

It binds agents, model adapter, tools, policy, checkpointer, store, and workspace
configuration into something executable. It is not a business concept and should
not define memory semantics by itself.

### Thread

Thread is continuity across runs.

It groups related runs that belong to the same ongoing task chain: resume,
follow-up, retry, delegated work, or multi-step collaboration.

The prior term `conversation` is chat-domain language. New conceptual docs
should prefer `thread`.

### Run

Run is one execution attempt.

A run has input, runtime state, run files, trace events, and a terminal status.
Child agents may create child runs. A thread may contain multiple runs.

### Store

Store is the persistence substrate.

It may be local files, SQLite, object storage, or another backend. Store is an
implementation capability, not the developer's primary mental model.

Store persists run files, memory records, trace files, checkpoints, and related
metadata.

### Context

Context is model input for one step.

It is assembled at runtime and is not a durable resource by itself. It may
include instructions, current task, recent messages, selected memory, selected
workspace references, tool descriptions, and output contract.

Context must not blindly include all memory, all workspace files, or trace.

### Memory

Memory is a compact reusable record that may be selected into future context.

It answers: should future runs possibly see this information again?

Memory is not trace, not a draft, not an artifact, not raw source material, and
not a full research result. Large or task-specific materials belong in run files
or external systems; memory may store a compact pointer.

Preferred conceptual scopes:

```text
user       cross-workspace user preference
workspace current work boundary facts or rules
agent      reusable method for an agent
thread     short-term continuity inside one task chain
```

Existing code may still use `project` and `conversation` for compatibility.
Documentation should describe them as legacy aliases for `workspace` and
`thread` until APIs migrate.

### Trace

Trace is append-only event history.

It answers: what happened during the run?

Trace records runtime facts such as run start/end, context build, model call,
tool call, policy decision, memory selection, memory write, and errors. Trace
does not decide behavior and does not enter context unless an application
explicitly converts trace into memory or task input.

## 4. Lifecycle

```text
User/App chooses Workspace
App creates Session
App creates or reuses Thread
Session starts Run
Runtime builds Context for each model step
Model and tools act
Runtime writes run files through Store
Runtime emits Trace
Reusable compact knowledge may become Memory
Run ends
Future runs may select Memory into Context
```

## 5. Creation Rules

| Concept | Producer | Timing | Notes |
|---|---|---|---|
| Workspace | User or application; Harness temporary fallback | Before session/run | Explicit is preferred. Temporary fallback lives under `./.modi/workspaces/<id>/`. |
| Session | Application/developer | Runtime setup | Binds agents/model/tools/policy/store. |
| Thread | Application or Harness fallback | Task start/continue | Reused for continuation. |
| Run | Session/runtime | Each execution | Owns run files and trace. |
| Store | Application/session config | Setup | Backend, not business concept. |
| Context | Runtime | Each model step | Temporary model input. |
| Memory | User/app, model proposal, or future summarizer | Before/during/after run | Must be compact and reusable. |
| Trace | Runtime | During run | Append-only events. |

## 6. Use Rules

| Question | Concept |
|---|---|
| Where does this work belong? | Workspace |
| Is this one execution attempt? | Run |
| Does this continue the same task chain? | Thread |
| Which runtime components are bound together? | Session |
| What does the model see now? | Context |
| Should future runs be able to reuse this? | Memory |
| What happened? | Trace |
| How is it persisted? | Store |

## 7. Workspace And Run Files

Workspace is the boundary. Run files are Harness-managed files for one run
inside that boundary.

Conceptual layout:

```text
<workspace>/
└── .modi/
    ├── runs/<run_id>/
    │   ├── input/
    │   ├── state/
    │   ├── refs/
    │   ├── drafts/
    │   ├── artifacts/
    │   └── logs/
    └── workspaces/<temporary_workspace_id>/   # only for fallback workspaces
```

Drafts and artifacts are not top-level architecture concepts:

- `draft`: intermediate run file.
- `artifact`: deliverable run file.
- `log`: run file containing logs; `trace.jsonl` is the standard trace stream.

## 8. Naming Migration Guidance

No public rename lands in V0.6.b, but new docs should use:

| Current / legacy | Preferred concept |
|---|---|
| `project` | `workspace` |
| `project_root` | `workspace_root` or `hook_root`, depending on use |
| `project_key` | `workspace_key` |
| `conversation` | `thread` |
| `WorkspaceManager` | run files manager / run store, conceptually |
| `workspace_root` as run storage root | `run_store_root`, conceptually |

The implementation can migrate gradually with aliases and compatibility tests.

## 9. Acceptance Criteria

- Core docs define Workspace, Session, Thread, Run, Store, Context, Memory, and
  Trace consistently.
- Workspace docs distinguish work boundary from run files.
- Memory docs define memory as compact reusable future context, not trace,
  draft, artifact, or raw task output.
- Context docs state that selected memory is part of model input but is not an
  instruction override.
- Trace docs state that trace is event history and does not enter context by
  default.
- Changelog records V0.6.b as a documentation and architecture alignment
  release.
- No runtime behavior changes are required.

## 10. Risks

- Existing code and older docs still use `project`, `conversation`, and
  `workspace_root`. V0.6.b should document compatibility instead of pretending
  those names disappeared.
- Overcorrecting with new abstractions would increase complexity. V0.6.b should
  stay vocabulary-first.
- Future API renames need migration tests and deprecation periods; they are out
  of scope for this release.
