# V0.5 — Three-Object Architecture: ModiHarness × ModiAgent × ModiSession

**Status:** Design approved 2026-06-03. Awaiting implementation plan.
**Breaking change:** Yes. V0.5 reshapes the public API.
**Predecessor:** V0.4.x — single God-Object `ModiHarness` holding everything from registries to compiled graph to thread metadata.

---

## 1. Motivation

V0.4.x ships a single class, `ModiHarness`, that simultaneously plays the roles
of: configuration container, registry of agents/tools/hooks, infra binder
(checkpointer, workspace, memory roots), graph compiler, runtime executor,
introspection facade, and thread index. Each `modi run` re-instantiates the
whole thing. The CLI and embedded examples each construct their own copy with
no shared concept of "what is the long-lived object." The string `agent="..."`
appears in every entry point but never corresponds to an in-memory object.

Three concrete consequences:

1. **"Harness shouldn't be instantiated"** — the current `ModiHarness` is heavy,
   re-built per CLI invocation, and conflates "what I declare" with "what is
   running."
2. **"Agent is not a runtime variable"** — agents are markdown files that
   become `AgentProfile` TypedDicts that float through `AgentState`; they have
   no first-class object, yet the API leaks `agents_dir=...` as if the harness
   owned them.
3. **"CLI is bewildering"** — three independent entry paths
   (`__main__._cmd_run`, `cli/runner.run_streaming`, `examples/.../run.py`),
   each builds its own harness, and `runner.py` reaches into
   `harness._agent_loader` to scrape what it needs.

The symptom is poor concept hygiene. The cause is missing layering: modi
mixes harness-layer concerns (model-era governance) with infra-layer concerns
(file roots, checkpointer backends) and application-layer concerns (which
agents exist, where they live).

## 2. Positioning

modi-harness is a **harness layer** for governed agents on top of LangChain +
LangGraph. It is one layer in a five-layer stack:

| Layer | Content | modi's relationship |
|---|---|---|
| Infra | checkpointer backends, filesystem, LLM provider clients, observability backends | modi **does not create or manage**; injected by callers |
| Runtime | LangGraph graph engine, state machine, checkpoint scheduler, interrupt/resume | modi **does not rewrite or abstract**; uses directly |
| **Harness ★** | Capabilities that put a model under reins: policy, hooks, context management, output contracts, skills, tool protocol, subagent delegation, trace, workspace, memory | **modi-harness itself** |
| Application Framework | Products like Claude Code, Codex, vertical agent platforms — built **on top of** modi-harness | modi's **downstream users** |
| Application | End-user agent products | modi does not touch |

**Design intent:** modi-harness is the substrate for the next Claude Code, the
next Codex, or a domain-specific agent platform. Its consumers are framework
authors, not end-users. The harness layer evolves with model generations
(Claude 4 → 5 → ...): prompt-caching APIs, tool-use protocols, structured
output, skills systems all change at the harness layer. Application frameworks
built on modi should not have to be rewritten when this happens — the harness
layer absorbs the churn behind a stable conceptual contract (Agent, Skill,
Tool, Policy, Memory, Workspace, Trace).

**Non-goals for modi-harness:**

- Not an application; does not serve end-users directly.
- Does not rewrite LangGraph runtime.
- Does not own infra-resource lifecycles.
- Does not preset application verbs (no `agent.run`, no `chat.send`, no
  ambient session, no module-level singleton).

## 3. Three Top-Level Objects

The V0.5 public API is exactly three classes plus supporting types.

### 3.1 `ModiHarness` — capability suite

A frozen, declarative description of "which model, governed by which
policies, hooks, output contracts." It does not know about specific agents.

```python
ModiHarness(
    chat_model: BaseChatModel,                  # required, injected
    *,
    rule_packs: list[str] | None = None,
    permissions: PermissionsConfig | None = None,
    hook_specs: list[HookSpec] | None = None,   # declarations, no execution
    builtin_tools: list[str] | None = None,     # whitelist of kernel tools
    model_cache: ModelAdapterCache | None = None,
)
```

**Holds:**

- `ModelAdapter` / `ModelAdapterCache` (wrapping the injected model client)
- `PolicyGate` (stateless)
- `HookRegistry` (declarations only — no dispatcher)
- `ContextManager`, `OutputController` (stateless policy objects)
- Builtin tool registry: kernel-scoped tools such as `submit_output`,
  `fetch_url`

**Does not hold:**

- Any specific `ModiAgent`
- Any `Skill` instance
- User business tools
- Any infra: checkpointer, workspace root, memory root
- `HookDispatcher`, `TraceMiddleware`, compiled graph
- Thread metadata

**Properties:** Immutable after construction. Shareable across many
`ModiSession`s. A process may construct multiple `ModiHarness` instances
(distinct configs, tenants, A/B variants).

**Lifecycle:** Constructed at application startup. Discarded at process exit.
Reconstructed when configuration changes; consumers swap the reference.

### 3.2 `ModiAgent` — declaration of a governable agent

A complete, self-contained, immutable definition of one agent: profile,
agent-scoped tools, skills, subagents, optional model override.

```python
ModiAgent(
    name: str,
    description: str,
    instruction: str,
    *,
    tools: list[ToolBinding] | None = None,         # agent-scoped tools
    skills: list[Skill] | None = None,
    subagents: list[ModiAgent] | None = None,       # recursive
    output_contract: OutputContract | None = None,
    permission_profile: PermissionProfile | None = None,
    safety_constraints: list[str] | None = None,
    model_override: ModelSpec | None = None,
    metadata: dict | None = None,
)

# Factory methods
ModiAgent.from_markdown(
    path: Path, *,
    tools=None, skills=None, subagents=None,
) -> ModiAgent

ModiAgent.load_dir(directory: Path) -> list[ModiAgent]
```

**Properties:**

1. **Self-contained** — does not depend on living in a registered directory.
2. **Code- or file-constructed, equivalent** — both produce the same kind
   of object.
3. **Recursively composable** — `subagents: list[ModiAgent]`.
4. **Immutable, not subclassable** — modify by composing tools/skills/policy,
   not by inheriting. Subclassing would re-open the governance boundary modi
   exists to enforce.
5. **No `run` method** — execution is exclusively on `ModiSession`. Agents are
   declarations, not executors. (This matches the chosen "Posture A":
   `session.run_task(agent="...", input=...)` is the sole execution entry
   point.)

**Tool / skill scoping:** Tools attached to a `ModiAgent` are visible only to
that agent (and to its declared subagents when explicitly passed in). They do
not pollute other agents.

### 3.3 `ModiSession` — `ModiHarness × ModiAgent × infra`

The binding object that combines a `ModiHarness` capability suite, a set of
`ModiAgent` declarations, and infra backends into something that can actually
execute threads.

```python
ModiSession(
    harness: ModiHarness,                        # held by reference
    *,
    agents: list[ModiAgent],                     # which agents are registered
    checkpointer: BaseCheckpointSaver,           # injected infra
    workspace_root: Path | str,
    memory_root: Path | str,
    project_root: Path | str | None = None,      # for hook dispatcher
    hook_pass_env: list[str] | None = None,
    max_steps: int = 20,
    repair_budget: int = 3,
)
```

**Holds:**

- Reference to its `ModiHarness`
- Indexed `dict[str, ModiAgent]` of registered agents
- `WorkspaceManager` (bound to `workspace_root`)
- `MemoryStore` (bound to `memory_root`)
- `HookDispatcher` (bound to `project_root` + `hook_pass_env`; runs the
  harness's `HookRegistry`)
- `TraceMiddleware` (bound to workspace, persists trace to disk)
- `ToolGateway` (merges harness builtin tools + each agent's scoped tools)
- Compiled LangGraph graph (built once at construction, immutable)
- Thread-metadata index (transitional in-memory dict; V0.5+ may move to a
  checkpointer-backed query)

**Public API:**

```text
# Execution
session.run_task(agent: str, input: dict, **opts) -> RunTaskResponse
session.resume_task(thread_id: str, payload: dict | None = None) -> RunTaskResponse
session.approve_action(thread_id, approval_id, decision="approved") -> RunTaskResponse
session.reject_action(thread_id, approval_id, reason) -> RunTaskResponse
session.stream(agent, input, **opts) -> Iterable[StreamEvent]
session.astream(agent, input, **opts) -> AsyncIterator[StreamEvent]

# Introspection (keyed by thread_id)
session.get_state(thread_id) -> AgentState | None
session.get_artifacts(thread_id) -> list[WorkspaceRef]
session.get_trace(thread_id) -> Iterable[TraceEvent]
session.get_denials(thread_id) -> list[DeniedAction]

# Memory / hooks / threads
session.add_memory / list_memory / forget_memory
session.list_hooks / get_hook_results
session.list_threads / end_thread

# Read-only agent lookup
session.get_agent(name) -> ModiAgent
session.list_agents() -> list[str]

# Resource cleanup
session.close()  # releases dispatcher subprocesses, trace handles, etc.
```

**Invariants:**

1. Does not create infra. Checkpointer, model client, file roots are all
   injected.
2. Compiles its graph at construction; the graph is then immutable. Adding
   agents/tools requires constructing a new `ModiSession`.
3. One `ModiHarness` may produce many `ModiSession`s (e.g., `MemorySaver` for
   tests, `PostgresSaver` for production, both pointing at the same harness).
4. Concurrency safety inherits from LangGraph + the injected checkpointer.
   `ModiSession` itself adds no locks.
5. `close()` releases modi-owned resources. Caller-owned infra (checkpointer
   connection, model client) is the caller's responsibility.

```

```

## 4. Data and Execution Flow

### 4.1 End-to-end `run_task`

```
1. Caller:
   session.run_task(agent="research-assistant", input={...}, mode="auto")

2. ModiSession:
   - Validate the agent name is registered.
   - Generate thread_id / run_id if not provided.
   - Pack harness + agents + bound deps into a GraphDeps view.
   - Delegate to GraphHarness (the renamed RuntimeAdapter).

3. GraphHarness:
   - Reuse or compile the LangGraph graph with GraphDeps + checkpointer.
   - Seed AgentState (agent_name, task, permission_mode, etc.).
   - Invoke graph.invoke(state, config={configurable: {thread_id, deps}}).

4. LangGraph runtime (not modi):
   - Drives nodes: model_turn → tool_dispatch → output_validate → ...
   - Each node pulls deps from config.
   - Checkpoint after each super-step.

5. Graph nodes (modi harness logic):
   - model_turn_node:    uses harness.model_adapter
   - tool_dispatch_node: routes through session.tool_gateway → policy gate → hook dispatcher → execution
   - output_validate:    uses harness.output_controller
   - hooks_dispatch:     uses session.hook_dispatcher to spawn hook subprocesses

6. GraphHarness:
   - Flush trace events through session.workspace.
   - Translate final state → RunTaskResponse.

7. ModiSession:
   - Touch thread metadata (last_active_at, run_count).
   - Return RunTaskResponse.
```

**Key:** `GraphDeps` is the merged view passed to nodes. Nodes never see
`ModiHarness` or `ModiSession` directly. The `ModiAgent` first-class concept
lives only at the modi public surface; inside the graph, agent data is
queried by name from `deps.agents[name]`.

### 4.2 Tool resolution: two scopes

```
ToolGateway (compiled once at ModiSession construction)
  ├─ kernel-scoped: harness.builtin_tools  (submit_output, fetch_url, ...)
  └─ agent-scoped: each ModiAgent's tools  (filtered by current agent)
```

When agent A delegates to subagent B, the graph switches the current-agent
context. B sees `harness.builtin_tools + B.tools`, never `A.tools`.

### 4.3 Hooks: declaration vs execution

```
HookRegistry (Harness, declares which event triggers which command)
        │ injected at construction
        ▼
HookDispatcher (Session, spawns subprocesses, enforces timeouts,
                bound to project_root + pass_env)
```

### 4.4 Interrupt / resume

```
run_task triggers approval_request:
   → graph.tool_dispatch_node fires LangGraph interrupt
   → run_task returns RunTaskResponse(status="interrupted", pending_approval=...)
   → caller decides
   → session.approve_action(...) or session.reject_action(...)
   → GraphHarness invokes graph.invoke(Command(resume=...), config={thread_id})
   → checkpointer reloads state, graph continues
   → fresh RunTaskResponse returned
```

Cross-process resume works identically when the checkpointer is durable
(Postgres / SQLite / Redis): a new process constructs a fresh `ModiSession`,
calls `resume_task(thread_id)`, and recovers the thread.

### 4.5 Persistence boundaries

| State | Storage | Cross-Session? | Cross-Process? |
|---|---|---|---|
| Thread state, messages, tool calls | checkpointer | yes (same checkpointer) | yes (durable backend) |
| Workspace artifacts | `workspace_root` filesystem | yes (same root) | yes |
| Memory records | `memory_root` filesystem | yes (same root) | yes |
| Trace events | `workspace_root/runs/<run_id>/logs/trace.jsonl` | yes (same root) | yes |
| `ModiHarness` instance | in-process | no | no (rebuilt) |
| `ModiSession` instance | in-process | no | no (rebuilt) |
| Session's local thread-metadata dict | in-process | no | no (transitional) |

After a server restart, a new process constructs a new harness and session
but reads back the same threads, workspace, memory, and traces by ID. This
preserves the CC/Codex durability model fully under the new layering.

### 4.6 Error flow

| Source | Where handled | Caller observes |
|---|---|---|
| `ModiAgent` config error | `ModiAgent.__init__` / `from_markdown` raises | construction-time exception |
| `ModiSession` config error (bad checkpointer, agent name conflict) | `ModiSession.__init__` raises | construction-time exception |
| Unregistered agent at run | `session.run_task` raises `AgentNotRegistered` | call-time exception |
| Policy denies tool call | inside graph → `denied_actions` | `RunTaskResponse(status="completed", denied_actions=[...])` |
| Hook denial / failure | dispatcher → graph | trace + possibly `status="blocked"` |
| Model invocation failure | model_adapter retry → final raise | `RunTaskResponse(status="failed", error=...)` |
| Output contract validation failure | output_validate → repair loop | after budget exhausted: `status="blocked"`, `draft_output` retained |
| Approval interrupt | LangGraph interrupt | `status="interrupted"`, `pending_approval` non-null |

**Key relocations:**

| Old | New | Note |
|---|---|---|
| `src/modi_harness/runtime/adapter.py` | `src/modi_harness/graph/harness_adapter.py` | rename + move; `runtime/` directory removed |
| `RuntimeAdapter` class | `GraphHarness` class | name now reflects "harness × graph glue" |


`GraphHarness`, `ToolRegistry`, `HookDispatcher`, etc. are internal and not

### 5.2 Plugin model

```python
# Old
PluginInfo = {
    "name": ...,
    "agents_dir": Path,           # directory coupling
    "skills_dir": Path,
    "tools": [(spec, handler), ...],
}

# New
PluginInfo = {
    "name": ...,
    "agents": list[ModiAgent],    # plugin parses its own markdown
    "builtin_tools": list[ToolBinding],   # only kernel-scoped tools here
}
```

Plugins now own their parsing. modi never crosses into plugin filesystem
directories.

## 6. Migration Plan

### 6.1 Steps (independently verifiable)

1. **Introduce `ModiAgent`.** New file `api/agent.py`. Upgrade
   `agents/loader.py` with `load_agent_object(path) → ModiAgent`. Add
   factory methods. Existing `load_agent → AgentProfile` retained but
   deprecated.
2. **Rename `RuntimeAdapter → GraphHarness` and relocate to `graph/`.**
   Pure rename + move. Remove `runtime/` directory. Update internal imports.
   Behavior unchanged.
3. **Split `ModiHarness` into the three objects.** Rewrite `api/harness.py`
   as the slim capability suite. Add `api/session.py` holding all infra
   bindings. Adapt `GraphDeps` to draw declarations from harness, backends
   from session. This is the heaviest step.
4. **Two-scope tool model.** `ModiHarness` holds the builtin registry;
   `ModiAgent` holds agent-scoped tools; `ModiSession` merges both into
   `ToolGateway`. `delegate_to_<name>` auto-registration moves to
   `ModiSession`, walking `ModiAgent.subagents`. Plugin `PluginInfo`
   reshaped.
5. **CLI adaptation.** `__main__.py` switches to the new posture:
   `ModiHarness(...) → ModiAgent.load_dir(...) → ModiSession(...) → session.run_task`.
   `cli/runner.py` accepts a `ModiSession` instead of a harness; remove
   `harness._agent_loader` access.
6. **Rewrite the three examples** under `examples/`. Update agent markdown
   if frontmatter schema changes.
7. **Documentation sync.** Rewrite `docs/architecture/08-harness-api.md`.
   Add or merge agent/session docs. Update README and example walkthroughs.
   Mark V0.5 breaking change.

### 6.2 Estimated impact

| Area | Files (estimated) |
|---|---|
| Library source | 15–18 |
| Tests | 20–30 (mostly fixture refactors) |
| Examples | 3 |
| Architecture docs | ~5 |
| Public API change | **`ModiHarness` constructor signature changes entirely** |

### 6.3 Breaking changes

V0.5 is intentionally breaking; no compatibility shim is provided.

1. `ModiHarness(agents_dir=..., workspace_root=..., checkpointer=..., chat_model=..., tools=...)`
   no longer works. Replaced by:
   - `ModiHarness(chat_model=..., ...)` (capabilities)
   - `ModiAgent.load_dir(agents_dir)` or explicit `ModiAgent(...)`
   - `ModiSession(harness, agents=[...], checkpointer=..., workspace_root=..., memory_root=...)`
2. `harness.run_task(...)` removed; use `session.run_task(...)`.
3. `harness.get_state` / `get_artifacts` / `get_trace` / `add_memory` etc. all
   move to `ModiSession`.
4. `harness._agent_loader` no longer exists.
5. `RuntimeAdapter` class name removed (was internal but documented).
6. Plugin `PluginInfo` shape changes (`agents_dir`/`skills_dir` →
   `agents: list[ModiAgent]`).

### 6.4 Test strategy

- Steps 1–2 should not break existing tests (additive + rename).
- Step 3 is the test-impact peak. All `ModiHarness` fixture construction
  changes to a two-stage `(ModiHarness, ModiSession)` pattern. Provide a
  test helper `make_session(...)` to encapsulate common fixtures.
- Add unit tests for `ModiAgent` and `ModiSession` covering: immutability,
  agent registration validation, recursive subagent construction,
  two-scope `ToolGateway` merge.
- Three examples passing with expected output is the acceptance line for
  steps 5–6.

### 6.5 Out of scope for V0.5

- Moving thread metadata from session-local dict to checkpointer-backed
  query (transitional dict retained; revisit V0.5+).
- HTTP API.
- Multi-tenant (left to application frameworks).
- Backwards-compatibility shim for old `PluginInfo`.
- Fine-grained `ModiSession.close()` resource cleanup beyond a basic
  release path.
- Splitting skills into a separate top-level concept (skills remain attached
  to `ModiAgent` for now).

### 6.6 Acceptance criteria

- All three examples run with consistent behavior.
- Existing test suite green after fixture refactor.
- New `ModiAgent` + `ModiSession` unit-test coverage ≥ 80%.
- `modi run` / `modi resume` CLI behavior unchanged from a user's point of
  view.
- `docs/architecture/08-*.md` rewritten to describe the three-object model.
- No file in `src/modi_harness/api/` exceeds 400 lines.

## 7. API Posture (decided during brainstorming)

Execution entry is **exclusively** on `ModiSession`:

```python
session.run_task(agent="research-assistant", input={...})
```

`ModiAgent` does not expose a `run` method. Application frameworks built on
modi may wrap this however they like (`agent.run`, `chat.send`,
`worker.assign`, ...) — modi does not preset the verb.

## 8. Naming Decisions

- Top-level class: `ModiHarness` (kept; only its responsibilities shrink).
- Second-level class: `ModiSession`.
- Third top-level class: `ModiAgent` (newly introduced).
- Internal graph adapter: `GraphHarness` (replaces `RuntimeAdapter`).
- `runtime/` directory: removed.

## 9. Embedded Usage Example (target)

```python
from modi_harness import ModiHarness, ModiAgent, ModiSession
from langgraph.checkpoint.memory import MemorySaver

# 1) Capability suite — knows nothing about specific agents.
harness = ModiHarness(
    chat_model=my_chat_model,
    rule_packs=["default"],
)

# 2) Agent declarations — markdown- or code-constructed, equivalent.
research = ModiAgent.from_markdown(
    "./agents/research-assistant.md",
    tools=[(FETCH_URL_SPEC, fetch_url)],
)

support = ModiAgent(
    name="support-bot",
    description="...",
    instruction="...",
    tools=[(spec, handler)],
    output_contract=...,
)

# 3) Session — binds harness, agents, and infra into something runnable.
session = ModiSession(
    harness=harness,
    agents=[research, support],
    checkpointer=MemorySaver(),
    workspace_root=".modi/workspace",
    memory_root="~/.modi/memory",
)

# 4) Execute.
response = session.run_task(
    agent="research-assistant",
    input={"goal": "...", "messages": [...]},
)
```
