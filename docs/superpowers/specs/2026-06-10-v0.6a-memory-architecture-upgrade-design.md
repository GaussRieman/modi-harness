# V0.6.a Design Spec — Governed Memory Architecture Upgrade

**Status:** implemented  
**Date:** 2026-06-10  
**Depends on:** V0.5 three-object architecture  
**Architecture reference:** [`docs/architecture/12-memory-store.md`](../../architecture/12-memory-store.md)  
**Implementation reference:** [`docs/implement/14-memory-store.md`](../../implement/14-memory-store.md)

---

## 1. Overview

V0.6.a upgrades Memory from a flat, rule-selected Markdown store into a governed long-term context subsystem.

The current implementation is valuable because it is simple and auditable, but it has four gaps:

1. Scope isolation is incomplete. `agent`, `project`, and `conversation` records are physically flat.
2. Stale records are stored but not filtered. `expires_at` and project horizons do not affect selection.
3. Retrieval is not explainable. Search returns records without scores or reasons.
4. Recalled memory is treated as trusted material by default. Selection does not distinguish instruction-level feedback from ordinary context.

V0.6.a keeps the Markdown ledger as the canonical source of truth and adds layered behavior around it.

---

## 2. Design Goals

- Preserve local, human-readable, auditable memory files.
- Add keyed physical isolation for user, agent, project, and conversation scopes.
- Keep legacy flat memory directories readable during migration.
- Filter expired and superseded memory from normal retrieval and context selection.
- Introduce explainable local retrieval candidates before adding vector or graph backends.
- Introduce a Memory Admission layer so recalled records are not automatically instruction-level authority.
- Route model-facing durable writes through proposal and policy.
- Add trace events that explain recall, admission, selection, and writes.

---

## 3. Non-Goals

- Do not replace the Markdown ledger with Mem0, Graphiti, Cognee, LangMem, or any external backend in V0.6.a.
- Do not introduce vector search in this version.
- Do not migrate or delete existing user memory files destructively.
- Do not remove public compatibility for `add_memory`, `list_memory`, `forget_memory`, `recall_memory`, or `save_memory`.
- Do not build an autonomous consolidation agent. Consolidation starts as safe report/index maintenance hooks.

---

## 4. Target Architecture

Memory has four responsibilities:

| Component | Responsibility |
|---|---|
| `MemoryLedger` | Canonical CRUD, scope paths, frontmatter files, audit metadata |
| `MemoryRetriever` | Candidate recall, scoring, reasons, retrieval signals |
| `MemoryAdmissionGate` | Context admission and authority classification |
| `MemoryConsolidator` | Index rebuild, dedupe reports, expiration and supersession maintenance |

The default implementation may keep these behind a compatibility `MemoryStore` facade, but the boundaries should be visible in code and tests.

---

## 5. Scope Keys

V0.6.a introduces `MemoryScopeKeys`:

```python
class MemoryScopeKeys:
    user_key: str
    agent_name: str
    project_key: str
    thread_id: str
```

Physical layout becomes:

```text
<memory_root>/
├── user/<user_key>/
├── agent/<agent_name>/
├── project/<project_key>/
└── conversation/<thread_id>/
```

Lookup remains ordered:

```text
conversation -> project -> agent -> user
```

Legacy flat directories remain readable as fallback:

```text
<memory_root>/agent/<record_id>.md
```

New writes with scope keys go to keyed directories.

---

## 6. Staleness and Supersession

Normal index/search/context paths omit:

- records whose `expires_at` is in the past
- records whose `metadata.superseded_by` is set
- project records older than `MODI_MEMORY_PROJECT_HORIZON_DAYS`

Explicit `read_record(id)` still returns expired or superseded records for audit and debugging.

Index/search APIs expose opt-ins:

```python
include_expired: bool = False
include_superseded: bool = False
```

---

## 7. Local Retrieval

V0.6.a introduces an internal `MemoryCandidate` shape:

```python
class MemoryCandidate(TypedDict):
    record: MemoryRecord
    score: float
    reasons: list[str]
    signals: dict[str, float]
```

The first local retriever remains lightweight:

- metadata filters
- substring compatibility
- deterministic scoring
- reasons for each recalled candidate

Vector, graph, and external backend adapters are deferred.

---

## 8. Memory Admission

V0.6.a introduces an internal `SelectedMemory` shape:

```python
class SelectedMemory(TypedDict):
    record: MemoryRecord
    authority: Literal["trusted", "context"]
    score: float
    reasons: list[str]
```

Admission filters expired, superseded, out-of-scope, low-confidence, and cross-domain candidates.

Authority rules:

- `trusted`: durable user feedback and approved project constraints relevant to the task.
- `context`: recalled facts, references, or background details that should not act as instructions.

Context rendering must preserve this distinction.

---

## 9. Write Lifecycle

Model-facing durable writes move toward a proposal flow:

```text
propose_memory
-> validate input
-> RequestedAction(kind="memory_write")
-> PolicyGate decision
-> commit allowed/approved writes
-> update indexes
-> trace event
```

Compatibility:

- `save_memory` remains available for `conversation` and `agent` scope writes during migration.
- Direct user API methods remain available and validate schema/source metadata.

Default policy:

- `conversation`: allow, subject to validation.
- `agent`: allow, subject to validation and duplicate checks.
- `project`: require approval.
- `user`: require approval.
- untrusted tool result source: deny unless user-confirmed or reviewed.

---

## 10. Trace Events

V0.6.a adds or prepares these events:

- `memory_recall_candidates`
- `memory_admission`
- `memory_selection`
- `memory_write_proposed`
- `memory_write`
- `memory_update`
- `memory_delete`
- `memory_consolidated`

Events should include record ids, scope keys, scores or reasons when available, and policy/admission decisions.

---

## 12. Acceptance Criteria

- Two agents, two project keys, and two thread ids cannot see each other's scoped records unless using shared higher scopes.
- Legacy flat memory directories remain readable.
- Normal context selection never injects expired or superseded memory.
- Explicit record reads can still retrieve expired/superseded records for audit.
- Retrieval candidates expose deterministic scores and reasons.
- Context memory preserves `trusted` vs `context` authority.
- Model-facing durable memory writes pass through Policy Gate.
- Trace can explain why memory was recalled, admitted, selected, written, updated, or deleted.
- Existing public API and builtin compatibility tests continue to pass.

---

## 13. Risks

- **Prompt behavior drift:** authority-aware memory rendering changes context shape and context hashes.
- **Migration ambiguity:** legacy fallback must not accidentally merge unrelated keyed records.
- **Policy bypass:** direct user API writes are intentionally privileged, but must still validate and audit source metadata.
- **Overbuilding retrieval:** local explainability comes before vector or graph integrations.
- **Timestamp semantics:** imported memory may preserve `updated_at`; `update_record` must still bump it.
