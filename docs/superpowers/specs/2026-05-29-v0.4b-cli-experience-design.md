# V0.4b Design Spec — Real CLI Experience

**Status:** draft
**Date:** 2026-05-29
**Depends on:** V0.3 (async streaming), V0.4a (model layer)

---

## 1. Overview

V0.4b makes the `modi` CLI feel like a real tool: live-streamed model output, colored tool activity, interactive approval prompts. Auto-detects TTY vs pipe so scripts still get clean JSON.

| Feature | Summary |
|---------|---------|
| Live streaming | Token-by-token rendering of `astream` events |
| Tool activity markers | Compact `▸ name(args)` / `← result` lines |
| Approval prompts | Inline panel + single-keypress decision |
| TTY auto-detect | Streams to TTY; emits JSON when piped |

---

## 2. Architecture

New `cli/` package under `src/modi_harness/` with three focused modules:

| Module | Responsibility |
|--------|----------------|
| `cli/renderer.py` | `StreamRenderer` — consumes stream events, renders via `rich` |
| `cli/prompt.py` | `ApprovalPrompt` — single-keypress inline approval flow |
| `cli/runner.py` | `run_streaming(harness, request)` — orchestrates astream + approvals |

The existing `__main__.py` is rewritten to:
1. Detect TTY via `sys.stdout.isatty()`
2. If TTY and command is `run` → call `cli.runner.run_streaming` (streaming)
3. If non-TTY (piped) or `--no-stream` flag → existing JSON dump path
4. `--stream` flag forces streaming even when non-TTY

---

## 3. Output Format

Streaming output for an interactive run:

```
[demo-agent] running...

I'll help you research that. Let me start by checking the docs.

▸ web_search(query="LangGraph 1.2 changelog")
← Found 3 results.

Based on the changelog, the main change is...

╭─ Approval needed ─────────────────────────────╮
│ Tool:  send_slack_release_summary             │
│ Args:  {"channel": "#releases", ...}          │
│ Risk:  L3                                     │
│ Reason: review_required                       │
╰───────────────────────────────────────────────╯
[a]pprove  [r]eject  [d]etails  > _

✓ completed in 8.3s
```

### 3.1 Color Conventions

| Element | Color |
|---------|-------|
| Model text | default (terminal foreground) |
| Tool markers (`▸ ←`) | cyan |
| Approval panel border | yellow |
| Status `completed` | green |
| Status `failed` / `rejected` | red |
| Status `interrupted` | yellow |
| Error messages | red |

### 3.2 Event → Render Mapping

| Event type | Render |
|------------|--------|
| `model_delta` | Append delta text inline (no newline) |
| `tool_call_proposal` | New line: `▸ <tool_name>(<args summary>)` |
| `tool_call_result` | New line: `← <result summary>` (truncated to 200 chars) |
| `approval_request` | Approval panel + prompt (see §4) |
| `terminal` | Status line: `<icon> <status> in <duration>s` |

Long tool args/results are truncated with `...` ellipsis. Full content available via `--verbose` flag (deferred to V0.4c if requested).

## 4. Interactive Approval Flow

When `approval_request` event arrives:

1. **Pause stream** — async lock prevents new events from rendering
2. **Print approval panel** — tool name, args (truncated to 80 chars), risk level, reason
3. **Prompt** — `[a]pprove [r]eject [d]etails > `
4. **Single keypress decision** via `rich.prompt.Prompt(choices=["a","r","d"])`
5. **On `a`**:
   - Call `harness.approve_action(thread_id, approval_id)`
   - Resume stream rendering with the new astream iterator
6. **On `r`**:
   - Prompt `Reason: ` (single line input)
   - Call `harness.reject_action(thread_id, approval_id, reason)`
   - Resume stream rendering
7. **On `d`**:
   - Print extended detail panel: full tool args, recent messages tail (last 3), denied_actions count, agent's safety_constraints
   - Re-prompt at step 3

Resume mechanic: after the user approves or rejects, the runner calls `harness.resume_task(thread_id, payload)` (non-streaming). The resulting messages and final output are rendered post-hoc — they appear at once rather than streaming token-by-token. This trade-off keeps the implementation simple. Token-level streaming after approval is deferred to V0.5.

## 5. TTY Detection and Mode Selection

```
Mode selection logic:
┌─ stdout is TTY? ─────┐
│  ├─ Yes              │
│  │   ├─ --no-stream? → JSON path
│  │   └─ default      → streaming path
│  └─ No (piped)       │
│      ├─ --stream?    → streaming path
│      └─ default      → JSON path
└──────────────────────┘
```

`sys.stdout.isatty()` is the source of truth. The `--stream` and `--no-stream` flags are mutually exclusive (argparse enforces).

## 6. New Dependencies

`pyproject.toml` adds:

No new dev dependencies. Existing `pytest-asyncio` covers async test paths.

## 7. Testing Strategy

| Test file | Coverage |
|-----------|----------|
| `tests/cli/test_renderer.py` | Feed mock events to `StreamRenderer`, capture via `rich.Console(record=True)`, assert markers/colors present |
| `tests/cli/test_prompt.py` | Mock stdin with scripted responses, verify approve/reject/details paths |
| `tests/cli/test_runner.py` | End-to-end: scripted model produces approval request, scripted user input approves, final status `completed` |
| `tests/cli/test_main.py` | TTY detection: stream path on TTY, JSON path on pipe |

The non-TTY JSON path preserves existing `__main__.py` behavior — current tests stay green.

## 9. Non-Goals

- Full TUI (split panes, scrollback, search)
- Stream-resume after approval (post-approval output appears post-hoc, not token-by-token)
- Persistent session shell (`modi shell` REPL — deferred)
- Configuration file for color themes
- Windows console-specific tweaks beyond what `rich` provides automatically
- Animated spinners during model thinking (deferred)
