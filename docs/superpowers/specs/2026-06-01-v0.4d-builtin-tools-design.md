# V0.4d — Builtin Tools (Workspace + Memory Primitives)

**Status:** design — pending implementation plan
**Author:** Frank Xiang
**Date:** 2026-06-01
**Predecessors:** V0.4c (plugin system), V0.4b (CLI experience)

## Goal

Eliminate boilerplate tool-registration code that every modi-harness consumer currently has to write to expose workspace and memory primitives. Make the six kernel-level primitives — `read_workspace_file`, `list_workspace_dir`, `save_artifact`, `save_draft`, `recall_memory`, `save_memory` — available to every agent by default, while keeping the existing governance pipeline (risk levels, permission modes, hooks, trace) fully in force.

This is **not** a port of the Claude Code builtin set. modi-harness is a domain-agnostic governance kernel; legal/research/devops/support agents need workspace and memory IO, not a filesystem `Bash` or a `WebFetch`. Domain-specific tools (file reads outside workspace, web access, third-party API calls) remain the consumer's responsibility, registered via `harness.register_tool(...)` or contributed via plugins.

## Non-goals

- File system access outside `.modi/workspace/<run_id>/`
- Network / web tools
- Subprocess / `Bash`-style tools
- Memory writes to the `user` scope (reserved for harness-level operations)
- Hot reload of builtin specs at runtime
- Per-tenant ACLs or quota systems

## Motivation

### The current pain

Every working example registers near-identical file IO tools:

```python
# examples/code_auditor/run.py
def read_file(path: str, max_lines: int = 200) -> dict: ...
LIST_PY_FILES_SPEC = { "name": "list_python_files", "risk_level": "L0", ... }
harness.register_tool(LIST_PY_FILES_SPEC, list_python_files)
harness.register_tool(READ_FILE_SPEC, read_file)

# docs/agents/research-assistant/agent.md
tools:
  - read_workspace_file       # caller must provide a handler
  - write_draft               # caller must provide a handler

# docs/agents/case-reviewer/agent.md
tools:
  - read_case
  - read_evidence
  - write_draft               # again
```

`write_draft` and `read_workspace_file` are mentioned in two separate sample agents with no shared implementation; in practice each consumer reinvents them. They are operations on **kernel-managed resources** (the run-scoped workspace) — there is no semantic reason for them to be domain-specific.

### Why workspace + memory specifically

The kernel already owns two resources that every agent uses regardless of domain:

| Resource | Manager | Cross-domain? |
|---|---|---|
| Workspace (run-scoped FS) | `WorkspaceManager` | Yes — every run produces artifacts and consumes inputs |
| Memory (cross-run facts) | `MemoryStore` | Yes — every domain has facts to remember |
| Subagent dispatch | `delegate_to_<X>` | Already auto-registered per-agent |
| Filesystem / web / DB / SaaS | Caller / plugin | No — domain-specific |

Builtin tools should mirror what the kernel already manages. The sandbox boundary is not "all `~/path` reads vs writes" — it is "the resources modi-harness owns vs the rest of the world."

## Design

### Builtin tool set (six tools)

| Tool | risk | side_effect | Behavior |
|---|---|---|---|
| `read_workspace_file` | L0 | false | Read text or bytes at `<run_id>/<kind>/<name>` |
| `list_workspace_dir` | L0 | false | List files under one workspace kind |
| `save_artifact` | L1 | true | Write `<run_id>/artifacts/<name>`, return new artifact_id |
| `save_draft` | L1 | true | Write `<run_id>/drafts/<name>` (overwrites) |
| `recall_memory` | L0 | false | Query `MemoryStore.search(scopes, types, tags, limit)` |
| `save_memory` | L1 | true | `MemoryStore.write_record(...)` — scope ∈ {`conversation`, `agent`} only |

Risk levels follow modi-harness convention: L0 = no side effect, L1 = local idempotent side effect. None of the six require approval in `auto`/`bypass` modes; in `ask` mode the L1 three trigger approval (existing `PolicyGate` behavior, not special-cased).

### Visibility model — implicit enable, trace-auditable

User decision (2026-06-01): builtins are **implicitly available** to every agent and **do not need to be listed** in `agent.md`'s `tools:` field. Audit happens via `trace.jsonl` — every call still produces a `tool_result` event with `tool_name`, `arguments`, and `decision`, so what an agent actually used is queryable post-hoc.

Rationale:

- agent.md's `tools:` field is intended to declare *domain* surface area. Workspace/memory primitives are not domain — they are kernel scaffolding. Forcing every agent to repeat them dilutes signal.
- `permission_profile.deny: [save_memory]` still works to opt out per-agent. Explicit deny beats implicit allow.
- Mode-based gating (`PolicyGate` + `risk_level`) is unchanged.

### `ToolSpec.kind` discriminant

```python
ToolKind = Literal["regular", "subagent", "builtin"]
```

`"builtin"` joins the two existing values. The discriminant drives two short-circuits in the gateway and one in the context manager.

### Context Manager change (`_resolve_visible_tools`)

Today (`src/modi_harness/context/manager.py:115`):

```python
agent_set = set(agent["default_tools"])
...
candidate = agent_set & skill_union if skill_union else agent_set
final = candidate & policy_visible
```

After:

```python
agent_set = set(agent["default_tools"])
builtin_set = {n for n, s in tool_catalog.items() if s["kind"] == "builtin"}
...
candidate = (agent_set & skill_union if skill_union else agent_set) | builtin_set
final = candidate & policy_visible
# ... existing subagent narrowing unchanged
```

`policy_visible` already filters by `permission_profile.deny`, so denying a builtin per-agent works out of the box.

### Tool Gateway changes (`execute_tool_call`)

Two surgical changes in `src/modi_harness/tools/gateway.py`:

1. **Visibility re-check** (line 131) becomes:

   ```python
   if spec["kind"] != "builtin" and tool_name not in agent["default_tools"]:
       return _error(...)
   ```

   Builtins skip the agent allowlist re-check (consistent with the context-manager change).

2. **Handler invocation** (line 241) becomes:

   ```python
   if spec["kind"] == "builtin":
       result_payload = entry.handler(
           arguments=args, state=state, deps=graph_deps,
       )
   elif state["permission_mode"] == "plan" and entry.dry_run is not None:
       result_payload = entry.dry_run(**args)
   else:
       result_payload = entry.handler(**args)
   ```

Everything else — schema validation, denied-retry, pre/post hooks, `PolicyGate.decide(...)`, idempotency cache, trust annotation, trace recording — runs unchanged. **Builtins are not a bypass for governance; they are a bypass for boilerplate.**

### Builtin handler signature

```python
BuiltinHandler = Callable[..., dict[str, Any]]
# Called as: handler(arguments=args_dict, state=state, deps=graph_deps)
```

Handlers receive the parsed arguments plus `state` (for `run_id`) and `deps` (for `WorkspaceManager` / `MemoryStore`). They never see `tool_call_id`, `agent_name`, or other governance metadata — that information stays in the gateway/trace layer.

This signature is **not** retrofitted onto regular tools — they keep `(**args)` for backwards compatibility with V0.4c plugins.

### Module layout

New file `src/modi_harness/tools/builtin.py`:

```python
# Public API
BUILTIN_TOOL_NAMES: frozenset[str] = frozenset({
    "read_workspace_file", "list_workspace_dir",
    "save_artifact", "save_draft", "recall_memory", "save_memory",
})

def get_builtin_specs() -> list[tuple[ToolSpec, BuiltinHandler]]: ...
```

Each handler is a top-level function (not a class) for testability. The six handlers and six specs live in this single file (~220 LOC estimated).

### `ModiHarness.__init__` changes

Two new parameters in `src/modi_harness/api/harness.py`:

```python
def __init__(
    self,
    *,
    ...,
    enable_builtin_tools: bool = True,
    builtin_tools: list[str] | None = None,
):
    ...
    if enable_builtin_tools:
        self._register_builtin_tools(only=builtin_tools)
```

- `enable_builtin_tools=False`: register zero builtins (existing tests that don't expect them keep passing).
- `builtin_tools=["save_artifact", "save_draft"]`: register only the listed subset.
- Default: register all six.

`_register_builtin_tools` is a thin wrapper that iterates `get_builtin_specs()` and calls `self._tools_registry.register_tool(spec, handler)` for each entry not filtered out.

### Sandbox enforcement

Each handler's safety comes from delegating to the existing safe-path infrastructure:

- `read_workspace_file(kind, name)` → `WorkspaceManager._safe_join(run_id, kind, name)`. Path traversal and absolute-path rejection are inherited.
- `save_artifact(name, content, mime_type=None)` → `WorkspaceManager.save_artifact(run_id, name, ...)`. The model **never specifies `run_id`** — it is injected from `state["run_id"]`. This is the structural sandbox: the model has no syntax to escape its own run.
- `save_memory(record)` validates `record.get("scope") in ("conversation", "agent")` before delegating to `MemoryStore.write_record(...)`. `user` scope is reserved for explicit `harness.add_memory(...)` calls.
- `recall_memory(scopes, types, tags, limit=20)` clamps `limit` to a max (e.g., 50) to prevent context blowup.

### Argument schemas

All six specs declare strict JSON schemas (`additionalProperties: false`, required fields). `Draft202012Validator` rejects malformed calls before execution.

Example for `read_workspace_file`:

```python
{
    "name": "read_workspace_file",
    "description": "Read a file from the current run's workspace.",
    "input_schema": {
        "type": "object",
        "properties": {
            "kind": {
                "type": "string",
                "enum": ["input", "state", "reference", "artifact", "draft", "log"],
            },
            "name": {"type": "string", "minLength": 1, "maxLength": 256},
            "encoding": {"type": "string", "enum": ["text", "bytes"], "default": "text"},
        },
        "required": ["kind", "name"],
        "additionalProperties": False,
    },
    "risk_level": "L0",
    "side_effect": False,
    "kind": "builtin",
}
```

The other six follow the same shape.

## Data flow (read_workspace_file example)

```
model emits tool_call("read_workspace_file", {"kind": "draft", "name": "summary.md"})
    ↓
graph.execute_tool_node
    ↓
ToolGateway.execute_tool_call(proposal, agent, state, graph_deps=deps)
    ├─ registry.get_entry("read_workspace_file")  →  spec.kind == "builtin"
    ├─ visibility re-check skipped (kind == builtin)
    ├─ schema validation
    ├─ denied-retry check
    ├─ pre_tool_use hooks
    ├─ PolicyGate.decide(...)  →  "allow" (L0, all modes)
    ├─ entry.handler(arguments=args, state=state, deps=deps)
    │      └─ workspace._safe_join(run_id, "draft", "summary.md").read_text(...)
    ├─ post_tool_use hooks
    └─ ToolDispatchResult(outcome="executed", record=..., trust=untrusted)
    ↓
trace event "tool_result" with {tool_name, arguments, decision: "allow", outcome: "executed"}
appended to <run_id>/logs/trace.jsonl
```

## Error handling

Each handler returns a result dict with one of:
- success shape (e.g., `{"path": ..., "content": ..., "size_bytes": ...}`)
- error shape `{"error": "<human readable reason>"}`

Hard failures (path outside workspace, missing run_id) raise — the gateway catches `Exception` (line 242) and surfaces as `outcome="error"`. This matches existing tool semantics.

`save_memory` with bad scope returns `{"error": "scope 'user' not writable from agent context"}` rather than raising — this is a model-correctable mistake, the model should adjust scope and retry.

## Testing

### Per-handler unit tests (`tests/tools/test_builtin.py`)

For each of the six tools:
1. **Happy path** — round-trip through `WorkspaceManager` / `MemoryStore` produces expected artifact / record / read content.
2. **Sandbox enforcement** — pass `kind=".."` or absolute path or traversal segment, expect rejection via existing `WorkspaceManager._safe_join` errors.
3. **Risk-respect** — for L1 builtins under `permission_mode=ask`, verify `outcome=interrupt` (not bypass).

### Integration test (`tests/tools/test_builtin_visibility.py`)

- An agent.md with `tools: []` (zero domain tools).
- Run a scripted model that calls `save_draft` and `read_workspace_file`.
- Assert: tool calls executed, draft file present on disk, trace contains both `tool_result` events.

### Subagent test

- Parent agent dispatches to a child agent, both rely on `save_artifact` builtin without listing it.
- Assert child's `save_artifact` lands in the child's `<child_run_id>/artifacts/` (run_id injection isolates child from parent).

### Existing-test compatibility

Existing tests don't list builtin names in their agent fixtures. With default `enable_builtin_tools=True`, those agents would suddenly see six extra tools in their context. Two mitigation paths:
- (a) Keep `enable_builtin_tools=True` default. Existing tests should not break because no scripted model in tests *calls* a builtin — they call domain tools the test fixture wires up. Worst case: visible-tools assertions become flaky if any test checks tool list size; we audit and adjust.
- (b) Tests use `enable_builtin_tools=False`. More invasive but safer.

Plan: **try (a) first**, run full suite, fall back to (b) if assertions break.

## Backwards compatibility

- Existing agents using `tools: [...]` keep working — the only change is that builtins are *additionally* visible.
- Existing `harness.register_tool(...)` calls work unchanged.
- Existing tests passing today should keep passing (per the (a) mitigation above).
- `code_auditor/run.py` is **not** rewritten. It remains the canonical example of "register a domain tool." The example documentation will mention "if you only need workspace IO, you can drop these registrations and rely on the builtins" as a footnote.

## Risks

- **Test surface drift.** Existing tests may assert on visible tool count and break. Mitigation: full-suite run during implementation; if breakage, follow plan (b).
- **Memory pollution via `save_memory`.** A buggy or adversarial agent could fill memory store with junk. Mitigation: scope restriction (`conversation`, `agent` only — `user` reserved for harness-level operations) plus `MemoryStore.write_record` already enforces structure validation.
- **Argument confusion.** Two ways to call workspace IO (builtin vs caller-registered) could confuse newcomers reading sample code. Mitigation: code_auditor stays as the "register your own" example; new `docs/builtins.md` explicitly contrasts the two paths.
- **Idempotency cache cross-talk.** Builtins go through the same `_idempotency_cache` as regular tools. `idempotent: false` is the default in `_DEFAULTS`, so unless a builtin opts in, no cross-talk. We will leave all six at `idempotent: false`.

## Out of scope (revisit later)

- Builtin `delete_artifact` / `delete_draft` — destructive, deserves its own design pass on confirmation flow.
- Builtin `web_search` / `web_fetch` — domain-y enough to ship as an optional plugin, not core.
- Builtin `delegate_with_isolation` — relates to subagent depth/policy, not this milestone.
- A `forget_memory` tool exposed to the model — same destructive concern as deletes.
- `read_artifact_by_id` deferred. Reason: `WorkspaceManager` has no artifact_id → path index. Model can re-read artifacts via `read_workspace_file(kind="artifact", name=...)` once `save_artifact` returns the `name`. Revisit when workspace gains an artifact registry.
