# V0.4c Design Spec — Plugin System

**Status:** shipped
**Shipped:** 2026-05-29
**Date:** 2026-05-29
**Depends on:** V0.4b (CLI experience)

---

## 1. Overview

V0.4c adds a plugin system to Modi Harness. Third parties can publish `pip install`-able packages that contribute agents, skills, and tools. Discovery uses Python entry points; failures are fail-fast.

| Feature | Summary |
|---------|---------|
| Entry point discovery | Plugins register via `modi_harness.plugins` entry point group |
| Single-function shape | One callable returns a dict with `name`, `agents_dir`, `skills_dir`, `tools` |
| Harness integration | `ModiHarness` auto-discovers and wires plugin contributions |
| `modi plugins list` | CLI subcommand inspects what's installed |
| Fail-fast errors | Broken plugins raise `PluginLoadError` immediately |

---

## 2. Entry Point Convention

Plugin authors declare in their `pyproject.toml`:

```toml
[project.entry-points."modi_harness.plugins"]
aws-tools = "modi_aws_tools:get_plugin"
```

The entry point points to a callable. When invoked, it returns a dict:

```python
def get_plugin() -> dict:
    return {
        "name": "aws-tools",
        "agents_dir": Path(__file__).parent / "agents",
        "skills_dir": Path(__file__).parent / "skills",
        "tools": [
            (tool_spec_dict, handler_callable),
            # ...
        ],
    }
```

All keys except `name` are optional. A plugin that contributes only tools omits `agents_dir`/`skills_dir`. A plugin that contributes only agents returns just `{"name": ..., "agents_dir": ...}`.

## 3. Discovery Module

New file `src/modi_harness/plugins.py`:

### 3.1 PluginInfo TypedDict

```python
class PluginInfo(TypedDict):
    name: str
    agents_dir: Path | None
    skills_dir: Path | None
    tools: list[tuple[dict, Callable]]
    source: str  # "entry_point:<dist_name> v<version>" or "explicit"
```

### 3.2 discover_plugins Function

```python
def discover_plugins() -> list[PluginInfo]:
    """Scan entry points group 'modi_harness.plugins'."""
```

Behavior:
1. Use `importlib.metadata.entry_points(group="modi_harness.plugins")`
2. For each entry point:
   - Load it (catches `ImportError`, `ModuleNotFoundError`, `AttributeError`)
   - Call the callable
   - Validate the returned dict (must have `name`, paths must exist if specified, tools must be valid spec dicts)
   - Construct `PluginInfo` with `source = f"entry_point:{ep.dist.name} v{ep.dist.version}"`
3. **Fail fast**: any error raises `PluginLoadError` with plugin name + source + original exception as `__cause__`
4. Return list in entry-point iteration order

### 3.3 PluginLoadError Exception

```python
class PluginLoadError(Exception):
    plugin_name: str
    source: str
    message: str
```

Error message format: `"Plugin '<name>' from <source> failed: <reason>"`.

### 3.4 Validation Rules

The dict returned by a plugin function must satisfy:
- `name` is a non-empty string
- If `agents_dir` is provided, it must exist as a directory
- If `skills_dir` is provided, it must exist as a directory
- If `tools` is provided, each entry must be `(dict, callable)` and the dict must have at minimum `name`, `description`, `input_schema`, `risk_level`

## 4. Harness Integration

`ModiHarness.__init__` gains two parameters:

```python
def __init__(
    self,
    *,
    # existing params...
    plugins: list[PluginInfo] | None = None,
    discover_plugins: bool = True,
    # ...
)
```

### 4.1 Discovery Logic

```
if plugins is None and discover_plugins:
    plugins = discover_plugins_fn()
elif plugins is None:
    plugins = []
```

### 4.2 Wiring to Loaders

Collect all `agents_dir` paths from plugins into a list. Pass to `AgentLoader(project_dir=agents_dir, plugin_dirs=collected_paths)`.

Same for `skills_dir` → `SkillLoader(plugin_dirs=...)`.

The `plugin_dirs` parameter is already supported by both loaders (V0.1) — we're feeding it now.

### 4.3 Tool Registration

After the harness's tool registry is constructed, iterate plugin tools:

```python
for plugin in plugins:
    for spec, handler in plugin.get("tools", []):
        self.register_tool(spec, handler)
```

If a plugin tool name collides with a previously registered tool, raise `ToolDuplicateError` (existing exception).

### 4.4 Subagent Tool Registration Order

`_register_subagent_tools()` runs AFTER plugin tool registration. Plugins can therefore introduce new agents whose `delegate_to_<name>` tools auto-register correctly.

## 5. CLI Integration

New subcommand `modi plugins list`:

### 5.1 Argparse

```
modi plugins list
```

No flags initially. Future: `--json` for machine-readable output.

### 5.2 Output Format

```
Discovered plugins:
  aws-tools (entry_point:modi-aws-tools v1.2.0)
    agents: 2 (aws-engineer, infra-reviewer)
    skills: 1 (aws-best-practices)
    tools:  5 (aws_describe_ec2, aws_list_s3, aws_get_logs, aws_invoke_lambda, aws_publish_sns)

  finance-pack (entry_point:modi-finance v0.3.1)
    agents: 1 (finance-analyst)
    tools:  3 (price_lookup, currency_convert, portfolio_value)

(2 plugins, 3 agents, 1 skill, 8 tools)
```

If no plugins are installed:
```
No plugins discovered.
Install plugins via: pip install <plugin-package>
See https://...  for the plugin author guide.
```

### 5.3 Implementation

`__main__.py` adds a `plugins` subparser. The handler calls `discover_plugins()`, prints the table. Errors during discovery print the `PluginLoadError` and exit code 1.

## 6. Test Plugin Fixture

For testing discovery without publishing real packages:

`tests/plugins/fixtures/sample_plugin/` — minimal package layout:
```
sample_plugin/
├── __init__.py          # exports get_plugin()
├── agents/
│   └── sample-agent.md
└── skills/
    └── sample-skill/
        └── SKILL.md
```

Tests use `monkeypatch` to inject a fake `entry_points()` result that returns an entry point pointing at the fixture's `get_plugin` function. This avoids requiring installed packages in the test environment.

For the failure cases, additional fixtures: `broken_plugin_no_name`, `broken_plugin_missing_dir`, `broken_plugin_raises_at_call`.

## 9. Non-Goals

- Plugin versioning / compatibility checks (plugin authors track Modi version themselves)
- Plugin sandboxing (plugins run with full process privileges)
- Hot-reload / dynamic plugin install
- Hook contribution (V0.5)
- Rule pack contribution (V0.5)
- Plugin marketplace or registry
- Per-plugin configuration in user settings (V0.5)
- Disabling individual plugins at runtime (uninstall the package instead)
