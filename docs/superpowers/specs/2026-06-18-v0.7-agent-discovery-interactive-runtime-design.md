# V0.7.0 Design Spec — Agent Discovery & Interactive Runtime

- Date: 2026-06-18
- Release: Modi Harness 0.7.0
- Theme: Agent Discovery & Interactive Runtime
- Status: proposed
- Implementation plan:
  [`2026-06-18-v0.7-agent-discovery-interactive-runtime-plan.md`](../plans/2026-06-18-v0.7-agent-discovery-interactive-runtime-plan.md)

## 1. Purpose

V0.7.0 turns Agents from objects that callers must manually assemble into
discoverable, inspectable, directly runnable project capabilities.

The release has two connected goals:

1. A user can enter a project and run a named Agent without knowing its file path:

   ```bash
   modi run research-assistant
   ```

2. Any opted-in Agent can expose a real task plan, pause for human review, and
   report truthful task transitions through a runtime protocol that every client
   can render consistently.

This replaces example-specific bootstrap scripts and task renderers with reusable
Harness capabilities while preserving the model-first posture from V0.6.d:
Harness owns discovery, state transitions, persistence, policy, and events; the
model still chooses the plan and performs the research.

## 2. Confirmed Current State

The design is grounded in the current repository:

- `modi run` exists, but requires `--agent`, `--agents-dir`, and `--task`.
- `ModiAgent.load_dir(...)` loads `*.md` and `<name>/agent.md` declarations.
- `ModiSession.from_discovery(...)` can merge one directory, plugin Agents, and
  explicit Agent objects, but there is no project locator or provenance registry.
- Installed plugins already contribute fully constructed `ModiAgent` objects.
- `examples/research_assistant/run.py` currently owns URL prompting, model setup,
  tool binding, plan review, task state validation, and a custom Rich renderer.
- LangGraph checkpoint interrupt/resume already works for governed tool approval.
- Stream events exist, but there are no canonical task-plan or user-interaction
  event types.
- Package metadata and `modi_harness.__version__` still say `0.5.0`; release
  metadata must be synchronized as part of V0.7.0.

## 3. First Principles

### 3.1 Agent does not know the CLI

An Agent never receives terminal styling, spinner state, viewport width, or Rich
objects. It sees only protocol tools and state supplied by Harness.

```text
Agent
  ↕ task / interaction protocol
Modi Harness
  ↕ canonical stream events
CLI | Web | Desktop | API
```

The same Agent must run unchanged behind a CLI, web UI, desktop app, or headless
API consumer.

### 3.2 Discovery must be explainable

No Agent may appear through unexplained import magic. Every discovered Agent has
source provenance, a stable qualified name, and an inspection command.

### 3.3 Task progress must be truthful

The renderer never invents progress and never marks a task complete on a timer.
Every visible transition comes from a validated runtime state transition.

### 3.4 Native planning is opt-in

Harness does not impose a fixed plan-execute-synthesize graph on every Agent.
Agents explicitly opt into the task protocol. Existing Agents remain behaviorally
unchanged unless they enable it.

### 3.5 Explicit configuration remains an escape hatch

`--agents-dir` remains supported. It becomes an optional explicit source instead
of a mandatory argument.

## 4. Goals

1. Discover Agents from project configuration, project conventions, installed
   plugins, user configuration, and explicit CLI directories.
2. Resolve a named Agent deterministically and expose its provenance.
3. Add `modi agents list/show/which`.
4. Support `modi run <agent-name>` while preserving old CLI syntax.
5. Introduce canonical task-plan state, protocol tools, stream events, and
   checkpoint persistence.
6. Add a real human plan-review interrupt that supports approve, revise, and
   cancel.
7. Add a generic CLI renderer with live task status plus append-only completion
   evidence.
8. Promote Research Assistant from an example-specific runner into the first
   project Agent package.

## 5. Non-Goals

- No hosted Agent marketplace.
- No remote registry or network installation.
- No GUI implementation.
- No autonomous scheduler, background daemon, or multi-run queue.
- No forced planning for Agents that do not opt in.
- No free-form natural-language command router.
- No arbitrary automatic execution of Python files from untrusted directories.
- No removal of `ModiAgent.from_markdown`, `load_dir`, or legacy CLI flags.

## 6. Release Shape

The implementation may land behind pre-release milestones, but the public release
is one coherent minor version:

- `0.7.0a`: project discovery, registry, provenance, and Agent CLI commands.
- `0.7.0b`: native task protocol, interaction interrupt, and generic renderer.
- `0.7.0`: Research Assistant promotion, compatibility validation, docs, and
  synchronized version metadata.

These labels are planning milestones, not a requirement to publish packages for
each pre-release.

## 7. Project Discovery

### 7.1 Project root

Starting at the current working directory, `ProjectLocator` walks upward until it
finds `modi.toml`. The containing directory becomes `project_root`.

If no `modi.toml` exists:

- `project_root` is the current working directory;
- conventional project directories are still considered;
- no configuration error is raised.

The walk stops at the filesystem root. Symlinks are resolved before deduplication.

### 7.2 `modi.toml`

V0.7.0 uses the standard-library `tomllib`; no new parser dependency is added.

```toml
[project]
name = "modi-harness"

[agents]
dirs = ["agents"]
include_conventional = true
include_plugins = true
include_user = true
trusted_project_factories = true
```

Rules:

- Relative paths resolve against `project_root`, never the process launch path.
- Unknown keys are errors with line/key context when available.
- Missing configured directories are errors; missing conventional directories are
  ignored.
- Configuration is read-only at runtime. `modi` does not rewrite `modi.toml`.

### 7.3 Discovery sources

Without `--agents-dir`, the registry considers:

1. Directories listed in `[agents].dirs`.
2. Conventional project directories:
   - `<project_root>/agents`
   - `<project_root>/.modi/agents`
3. Installed `modi_harness.plugins` entry points.
4. User declarations in `~/.modi/agents` when enabled.

With one or more `--agents-dir PATH` arguments, each path is added as an explicit
source. An unqualified name found in an explicit source wins only because the user
provided that source on this command. All other candidates remain inspectable by
qualified name.

### 7.4 Supported source forms

#### Declarative Agent

Existing forms remain valid:

```text
agents/reviewer.md
agents/reviewer/agent.md
```

They load through `ModiAgent.from_markdown` / `load_dir` and cannot contribute
Python tool handlers by themselves.

#### Project Agent package

Mature examples often need tools and skills. A trusted project package uses:

```text
agents/research_assistant/
  agent.md
  agent.toml
  runtime.py
  skills/
```

`agent.toml`:

```toml
factory = "runtime:build_agent"
```

`build_agent()` returns one `ModiAgent`. The loader imports the factory relative
to the package directory and validates the return type and declared name.

Security rules:

- Python factories are loaded only from explicit sources or the active project's
  configured/conventional directories when `trusted_project_factories = true`.
- User-level convention scanning loads Markdown only in V0.7.0.
- Installed plugin factories continue to be trusted through Python package
  installation and entry points.
- `modi agents show` clearly marks executable project factories.

### 7.5 Provenance model

```python
AgentSourceKind = Literal["explicit", "project", "plugin", "user"]

AgentDescriptor:
    name: str
    qualified_name: str
    source_kind: AgentSourceKind
    source_id: str
    path: Path | None
    plugin_name: str | None
    executable_factory: bool
    agent: ModiAgent

DiscoveryResult:
    registry: AgentRegistry
    kernel_tools: list[ToolBinding]
    project_root: Path
    config_path: Path | None
```

Qualified names:

- `explicit:research-assistant`
- `project:research-assistant`
- `user:research-assistant`
- `plugin:<plugin-name>/research-assistant`

The registry exposes descriptors, not only `ModiAgent` values, so provenance is
not discarded during session construction. Discovery also retains plugin
`kernel_tools`; resolving an Agent must not accidentally discard the plugin
capabilities needed to construct its `ModiHarness` / `ModiSession`.

### 7.6 Resolution and conflicts

- One candidate for an unqualified name: resolve it.
- More than one distinct candidate: raise `AgentResolutionError` and list all
  qualified names.
- The same canonical path discovered twice: deduplicate.
- Two equal plugin/project Agent values from different sources are still separate
  descriptors; source identity matters.
- When `--agents-dir` was explicitly supplied and contains the requested
  unqualified name, select that explicit candidate and report shadowed candidates
  in verbose diagnostics.
- A qualified name always selects exactly that source or fails.

There is no silent project-vs-user or project-vs-plugin shadowing.

## 8. Agent Registry API

New module: `modi_harness.discovery`.

```python
result = discover_agents(
    cwd=Path.cwd(),
    explicit_dirs=[...],
    include_plugins=True,
    include_user=True,
)

registry = result.registry

registry.list() -> list[AgentDescriptor]
registry.resolve("research-assistant") -> AgentDescriptor
registry.resolve("project:research-assistant") -> AgentDescriptor
registry.explain("research-assistant") -> ResolutionReport
```

Responsibilities:

- project/config location;
- source scanning;
- factory loading under trust rules;
- provenance capture;
- conflict detection and resolution;
- deterministic ordering for output and tests.

It does not construct models, checkpointers, workspaces, memory stores, or
`ModiSession` objects. The returned `DiscoveryResult.kernel_tools` is passed to
`ModiHarness` by the CLI/session factory; the registry itself remains focused on
Agent identity and resolution.

## 9. CLI Surface

### 9.1 Agent inspection

```bash
modi agents list
modi agents list --verbose
modi agents show research-assistant
modi agents which research-assistant
modi agents which research-assistant --all
```

Expected output:

```text
research-assistant
  qualified: project:research-assistant
  source: project package
  path: /repo/agents/research_assistant
  factory: runtime:build_agent
  task protocol: required, review before execution
```

`list` never imports a user-level Python factory. Project factories may be loaded
only under the trust rule in §7.4.

### 9.2 Running an Agent

Preferred syntax:

```bash
modi run research-assistant
modi run project:research-assistant
modi run research-assistant --task task.json
echo '{"prompt":"hello"}' | modi run support-bot --task -
```

Compatibility syntax remains valid through V0.7.x:

```bash
modi run --agent research-assistant --agents-dir agents --task task.json
```

Rules:

- Positional Agent and `--agent` are mutually exclusive.
- `--agents-dir` becomes repeatable and optional.
- `--task` becomes optional only for a TTY.
- If `--task` is omitted in a TTY, CLI prompts once for the initial request.
- If `--task` is omitted on a non-TTY, exit with an actionable error rather than
  waiting on stdin implicitly.
- Existing `--stream` / `--no-stream` behavior remains compatible.

### 9.3 Initial interactive input

V0.7.0 does not build a generic form engine. The default is:

```text
Task for research-assistant
> Research DeepSeek API pricing using https://...
```

Agent-specific structured input can still come from `--task`. A richer input
contract is deferred until real promoted Agents demonstrate common field types.
Research Assistant may use the native user-interaction protocol after startup to
request URLs or clarify the question.

## 10. Native Task Protocol

### 10.1 Agent declaration

Agents opt in through frontmatter / `ModiAgent` metadata:

```yaml
task_protocol:
  mode: required          # off | optional | required
  review: before_execution # never | before_execution
  min_items: 1
  max_items: 8
```

Defaults are `mode: off`, `review: never`. Existing Agents receive no task tools
and no validation changes.

### 10.2 Canonical types

```python
TaskStatus = Literal["pending", "in_progress", "completed", "blocked"]

TaskItem:
    id: str
    title: str
    status: TaskStatus
    summary: str | None

TaskPlan:
    version: int
    items: list[TaskItem]
    current_task_id: str | None
    current_action: str | None
    last_activity: str | None

PendingInteraction:
    interaction_id: str
    kind: Literal["plan_review", "user_input"]
    prompt: str
    payload: dict
```

`TaskPlan` and `PendingInteraction` live in `AgentState`, so checkpoint resume and
cross-process inspection preserve them.

### 10.3 Protocol tools

When task protocol is enabled, Harness injects protocol tools:

```text
create_task_plan(tasks)
revise_task_plan(tasks, reason)
start_task(task_id, current_action)
resume_task(task_id, current_action)
complete_task(task_id, summary, next_task_id?, current_action?)
block_task(task_id, reason)
```

The tools are implemented by Harness, not supplied by each Agent. They update
canonical state and emit events. They do not expose renderer concepts.

Transition invariants:

- IDs and order are stable after plan approval; revision creates a new version.
- At most one task is `in_progress`.
- `pending -> in_progress -> completed|blocked`.
- `blocked -> in_progress` only through explicit `resume_task` after the blocker changes.
- Completed tasks never regress.
- `complete_task` completes exactly one task.
- `next_task_id`, when supplied, starts exactly one pending task atomically.
- `submit_output` is rejected while a required plan has unfinished tasks.
- A plan update and `submit_output` cannot be accepted as sibling calls in one
  model turn.

### 10.4 Model-first boundary

Harness validates transitions but does not generate task titles, decide task count,
or infer completion from tool calls. The model creates and advances the plan.

## 11. Human Plan Review

### 11.1 Interaction, not policy approval

Plan review is a first-class interaction interrupt. It must not masquerade as an
L3 side-effect approval.

After `create_task_plan` with `review: before_execution`:

1. The proposed plan is stored in checkpoint state as pending.
2. Graph calls LangGraph `interrupt(...)` with `kind="plan_review"`.
3. A client displays the plan and collects one of:
   - approve;
   - revise with feedback;
   - cancel.
4. `Command(resume=...)` resumes the same thread.

Resume payloads:

```json
{"interaction_id":"...","decision":"approved"}
{"interaction_id":"...","decision":"revise","feedback":"add cost examples"}
{"interaction_id":"...","decision":"cancelled"}
```

On revision, the model receives feedback and must call `revise_task_plan`; the
revised plan interrupts again. On cancellation, the run ends `cancelled` without
executing research tools.

### 11.2 Graph integration

Plan review uses a dedicated graph node so pending state is checkpointed before
LangGraph interrupts:

```text
model_turn
  -> execute_tool          # create/revise plan validates and stores pending state
  -> await_interaction     # interrupt() and apply Command(resume=...)
  -> model_turn
```

`execute_tool` does not call `interrupt()` for task protocol operations. It writes:

- the proposed `TaskPlan`;
- `pending_interaction` including the originating protocol tool call id;
- task/interaction trace events.

`route_after_tool` sends the committed state to `await_interaction`.
`await_interaction` calls `interrupt(...)`. After resume it validates
`interaction_id` and emits exactly one protocol tool-result message:

- approved: plan becomes active and execution may continue;
- revise: pending plan stays non-executable and the tool result carries feedback
  for the model;
- cancelled: run status becomes `cancelled`.

This keeps model tool-use/tool-result history well formed and prevents research
tools from executing in the same batch after an unreviewed plan call. Any sibling
calls after `create_task_plan` / `revise_task_plan` are deferred.

### 11.3 Generalization boundary

The state/event envelope supports `kind="user_input"`, but V0.7.0 only requires
the public API and CLI flow for plan review. Generic multi-question forms are not
required for release acceptance.

## 12. Canonical Stream Events

Add these `StreamEventType` values:

```text
task_plan_created
task_plan_revised
task_started
task_completed
task_blocked
interaction_request
interaction_resolved
```

Event payloads include `plan_version`, task identifiers, summaries, and the
complete current plan when needed. They never contain Rich markup.

Ordering guarantees:

- State is committed before its event is emitted.
- `task_completed(A)` precedes `task_started(B)` unless both are represented by
  one atomic `complete_task(..., next_task_id=B)` update; in that case one stream
  projection emits completed then started deterministically.
- `interaction_request` is terminal for the current stream segment.
- `sequence` is monotonic within one stream segment, matching the existing API.
- Every event also carries a stable `event_id`; resumed segments may restart
  `sequence` without creating ambiguous events.
- `segment_id` identifies the initial or resumed stream segment. Cross-segment
  consumers order durable task transitions by checkpoint/trace order and `event_id`,
  not by assuming one process-local integer survives resume.

Trace events mirror task and interaction transitions for auditability.

## 13. Generic CLI Task Renderer

### 13.1 Live TTY view

```text
Research plan · 2/4 complete
✓ 获取官方定价页面
✓ 提取模型价格差异
● 分析缓存折扣
○ 生成结构化简报

fetch_url  ✓  21 KB · 1 source
当前：核对缓存命中与未命中的成本差异
```

Behavior:

- Live checklist is driven only by task events.
- Current action uses a spinner while a task is active.
- Tool results are summarized using generic metadata when available; clients may
  fall back to the existing truncated representation.
- Final checklist remains visible.

### 13.2 Append-only evidence

Rich Live overwrites intermediate frames, so the renderer also emits one concise,
append-only line above the live region for every completed or blocked task:

```text
✓ 1/4 获取官方定价页面 — 读取 21 KB 官方文档
✓ 2/4 提取模型价格差异 — 找到 2 个模型档位
```

Copied terminal output and CI logs therefore retain real progress history.

### 13.3 Non-TTY behavior

- Existing default final JSON behavior remains unchanged.
- Add an explicit `--stream-format jsonl` for canonical event consumers.
- `--stream-format plain` emits append-only human-readable task/tool lines without
  ANSI or Live redraw.
- `--stream-format live` requires a TTY unless forced.

## 14. Session And API Surface

New or extended APIs:

```python
ModiSession.from_registry(...)
ModiSession.get_task_plan(thread_id) -> TaskPlan | None
ModiSession.respond_to_interaction(thread_id, interaction_id, payload)
ModiSession.astream_resume(thread_id, payload)  # retained/publicly documented
```

Existing `approve_action` / `reject_action` remain policy-specific compatibility
APIs. They are not reused for plan feedback.

`RunTaskResponse.status` adds `cancelled`, and returns `pending_interaction`
separately from `pending_approval`.

## 15. Research Assistant Promotion

Target layout:

```text
agents/research_assistant/
  agent.md
  agent.toml
  runtime.py
  skills/
```

Migration rules:

- Move Agent declaration and skills from `examples/research_assistant` into the
  project Agent package.
- Move `fetch_url` binding into `runtime.py` initially; a generic web-fetch builtin
  is a separate future decision.
- Remove example-specific task protocol tools and `ResearchTaskRenderer`.
- Native task protocol replaces `propose_research_plan` and
  `update_research_tasks`.
- Keep `examples/research_assistant/run.py` as a thin compatibility/demo wrapper
  that resolves and runs the project Agent; it must not own orchestration.
- Register `agents` in repository `modi.toml`.

Acceptance invocation:

```bash
modi agents which research-assistant
modi run research-assistant
```

## 16. Error Handling

| Case | Result |
|---|---|
| No Agent found | Exit 2; show searched sources and `modi agents list` hint |
| Ambiguous unqualified name | Exit 2; list qualified candidates |
| Invalid `modi.toml` | Exit 2; identify key/path and do not partially discover |
| Configured directory missing | Exit 2; conventional missing directory is ignored |
| Factory import/return failure | Mark source and factory in actionable error; chain cause |
| Untrusted user factory | Do not import; show as unsupported executable source |
| Invalid task transition | Tool result error; state unchanged; model may repair |
| Plan revision feedback | Resume same thread; request revised plan; interrupt again |
| User cancellation | Status `cancelled`; no research tools execute |
| Client disconnect during review | Checkpoint remains interrupted and resumable |
| Required plan incomplete at submit | Output validation rejects with actionable hint |

## 17. Compatibility

- Existing direct construction (`ModiHarness` + `ModiAgent` + `ModiSession`) remains
  supported.
- Existing `ModiSession.from_discovery(...)` remains and may delegate to registry
  internals without changing its current signature.
- Existing `modi run --agent ... --agents-dir ... --task ...` remains supported.
- Agents without `task_protocol` see no new tools or output restrictions.
- Existing stream event consumers must ignore unknown event types as today.
- Existing policy approval payloads and APIs remain distinct and compatible.

## 18. Security

- Resolve all discovery paths and prevent duplicate/surprising symlink traversal.
- Never auto-import Python from user convention directories.
- Treat Agent markdown and project configuration as trusted project code only after
  project selection; fetched/runtime content remains untrusted.
- Factory imports run in-process and are equivalent to importing project code;
  `agents show` must disclose this clearly.
- Qualified names prevent a user Agent from impersonating a project/plugin Agent.
- No network access occurs during discovery.

## 19. Observability

Trace additions:

- `agent_discovery_started`
- `agent_discovered`
- `agent_resolved`
- `task_plan_created`
- `task_plan_revised`
- `task_started`
- `task_completed`
- `task_blocked`
- `interaction_requested`
- `interaction_resolved`

Discovery CLI diagnostics must show provenance without requiring trace inspection.
Runtime task events and trace records use the same task IDs and plan versions.

## 20. Testing Strategy

### Discovery

- Project-root upward search and no-config fallback.
- Relative path resolution against project root.
- Configured, conventional, plugin, user, and explicit sources.
- Canonical-path deduplication.
- Ambiguous names and qualified resolution.
- Explicit directory selection behavior.
- Trusted project factory and rejected user factory.
- Deterministic listing and provenance output.

### CLI

- `agents list/show/which` output and exit codes.
- Positional Agent syntax and compatibility flags.
- TTY task prompt and non-TTY missing-task error.
- Live/plain/jsonl stream format selection.

### Task protocol

- Valid create/start/resume/complete/block transitions.
- Invalid skip, regression, duplicate active task, and multi-complete rejection.
- Required-plan submit guard.
- Task state survives checkpoint resume and process reconstruction.
- Canonical stream/trace event ordering.
- Step exhaustion ends as `failed` with `max_steps_exceeded`, never `running`.

### HITL

- Plan approve then execution.
- Revision feedback, revised plan, second interrupt, approval.
- Cancellation before any research tool.
- Disconnect/restart/resume from pending interaction.
- Policy approval and plan interaction remain independently addressable.

### Renderer

- Live checklist reflects canonical events only.
- Append-only completion lines survive captured output.
- Tool result summary avoids raw large dictionaries.
- Plain and JSONL modes contain every transition.

### Promotion

- Registry discovers the project Research Assistant package.
- `modi run research-assistant` reaches plan review and completes with scripted
  model/tool fixtures.
- Compatibility example wrapper uses registry/runtime rather than local task UI.

## 21. Acceptance Criteria

1. From repository root, `modi agents which research-assistant` resolves a project
   Agent with an inspectable path and factory.
2. `modi run research-assistant` works without `--agents-dir`.
3. Old explicit CLI syntax remains green.
4. An opted-in Agent creates a task plan through native protocol tools.
5. Plan review is a real checkpoint interaction supporting approve, revise, and
   cancel on the same thread.
6. Task state is checkpointed, transition-validated, streamable, traceable, and
   available through `ModiSession`.
7. Generic CLI displays live progress and preserves append-only completion history.
8. No example-specific task renderer is required for Research Assistant.
9. Agents without task protocol do not change behavior.
10. Full tests, ruff, and mypy pass; package and runtime versions report `0.7.0`.
