# V0.6.e Design Spec — Execution Efficiency (Batch Tool Execution + Recall Cache)

## 1. Purpose

V0.6.e is an **execution-layer efficiency** correction. It does not change what the
model decides; it removes two sources of wasted work that the model-first posture
(v0.6.d) makes visible:

1. **One-tool-per-turn serialization.** When the model issues several tool calls in
   one turn, `execute_tool_node` runs only the first and defers the rest, forcing the
   model to re-issue them across extra turns.
2. **Per-turn memory recall recomputation.** `model_turn_node` recomputes memory
   recall and selection on every turn with identical inputs.

Both push deterministic work back onto the model as extra turns. V0.6.e sinks that
deterministic work into the substrate, faithful to the first principle: Harness
exists to support model execution (reliable tools, reliable memory), extending model
capability rather than competing with model reasoning.

This is a **purely additive execution-layer change**. No runtime rewrite, no fixed
plan→execute orchestration (that would make Harness a planner — spec v0.6.d §3).

## 2. Background — Confirmed Root Causes

Grounded in the current code:

- **One tool per turn:** `src/modi_harness/graph/nodes.py:416` `execute_tool_node`
  executes `pending[0]` only; `nodes.py:509-523` emits synthetic `deferred`
  tool_results for `pending[1:]` and asks the model to re-issue them sequentially.
  A turn that issues two `fetch_url` calls becomes multiple turns.
- **Per-turn recall:** `src/modi_harness/graph/nodes.py:114-127` `model_turn_node`
  calls `recall_candidates_for_context()` and `select_for_context()` every turn. The
  task is stable within a run, so the same scoring runs N times.

The two compound: each deferred re-issue triggers another `model_turn`, which runs
another uncached recall.

The graph topology (`builder.py`): `START → setup → model_turn →
route_after_model → {execute_tool | validate_output}`, with `execute_tool` and
`validate_output` routing back to `model_turn`. **This topology does not change.**

## 3. Architectural Position

Two changes, both additive:

- **Batch tool execution** — change only `execute_tool_node` to process the entire
  `pending` list in one node visit (serially, deterministic side-effect order).
- **Recall cache** — a small in-process `RunRecallCache` on `GraphDeps`, consulted by
  `model_turn_node`, invalidated by `execute_tool_node` on memory writes.

Out of scope (deliberately): true concurrent execution, fixed orchestration,
structured-output hardening, preflight checks, observability metrics. These are the
other feedback items and are separate future iterations.

## 4. Feature A — Batch Tool Execution

### 4.1 Scope

Only `execute_tool_node` (`nodes.py:416`) changes. Graph structure, routing, and
`model_turn_node` are untouched.

### 4.2 Behavior

Process the full `pending` list in one node visit:

```text
results = []
for i, proposal in enumerate(pending):
    if proposal.malformed:
        results.append(_handle_malformed(...)); continue
    dispatch = deps.tools.execute_tool_call(proposal, ...)   # existing call, per item
    if dispatch.outcome == "interrupt" and dispatch.decision is not None:
        # Stop-at-approval (decision below):
        #   - already-executed results land in messages/tool_calls
        #   - this call raises the approval interrupt (existing _apply_resume_decision)
        #   - pending[i+1:] are marked deferred; re-issued next turn after resume
        emit interrupt; mark pending[i+1:] deferred; return
    results.append(dispatch)                                  # executed / error / denied_retry
# all calls cleared without approval → one tool_result message per call
return {
    "messages": [...one tool_result per result...],
    "tool_calls": [...records...],
    "pending_tool_calls": [],
    "pending_trace_events": [...per-call events...],
}
```

### 4.3 Invariants and Decisions

- **Protocol invariant:** every `tool_use` issued this turn gets exactly one
  `tool_result` (executed / error / denied_retry / deferred). Message history stays
  well-formed. This is the same constraint the current `deferred` logic guards; the
  difference is most calls now get a real result the same turn.
- **Stop-at-approval (chosen):** when a mid-batch call requires human approval, the
  already-executed results are committed, that call raises the interrupt via the
  existing `_apply_resume_decision` path, and `pending[i+1:]` are marked `deferred`
  for re-issue on the next turn after resume. Batches containing an approval may
  still take an extra turn; non-approval batches collapse to one.
- **Per-call error isolation (default):** a single call's error (e.g. a schema
  validation failure like `save_artifact({})`) produces only its own error
  tool_result and does **not** abort the batch; remaining calls still execute. The
  model sees all per-call results together next turn.
- **Serial side-effect order:** calls execute in `pending` order within the node (no
  concurrency), so workspace writes and memory writes have a deterministic order. No
  concurrency-safety concerns are introduced.
- **`step_count`:** one batched node visit still counts as one step (unchanged), so
  `max_steps` budget (`route_after_tool`, `nodes.py:782`) now covers more actual tool
  calls rather than tripping earlier.

## 5. Feature B — Memory Recall Cache

### 5.1 Component

A new small component, single responsibility: cache one run's recall+select result.

```text
class RunRecallCache:
    def get_or_compute(run_id, compute_fn) -> (recalled_candidates, selected_records)
    def invalidate(run_id) -> None
```

- In-process, keyed by `run_id`. Lives on `GraphDeps` (peer of `memory`, `context`).
- **Not** stored in LangGraph checkpoint state — not serialized.

### 5.2 Wiring

- **`model_turn_node` (`nodes.py:114-127`):** wrap the existing
  `recall_candidates_for_context()` + `select_for_context()` pair in
  `cache.get_or_compute(run_id, compute_fn)`. On hit, return the prior result; on
  miss, call the store as today. `build_context` and everything else unchanged.
- **`execute_tool_node`:** when a `memory_write` signal is detected — `save_memory`
  returned an `id`, or `propose_memory` status is `committed` (the existing
  detection at `nodes.py:722,748`) — call `cache.invalidate(run_id)`. The next
  `model_turn` misses and recomputes once, incorporating the new record.

### 5.3 Invalidation Semantics (chosen: invalidate-on-write)

- Cache is computed once per run and reused until a memory **write commits** in this
  run. The task is stable within a run, so a committed write is the materially
  relevant invalidation signal.
- `propose_memory` that only reaches `approval_required` or `denied` does **not**
  invalidate (the store is unchanged).
- **Cross-process resume:** the in-process cache is empty in a new process, so the
  first `model_turn` after resume recomputes once (cost = one recall, negligible),
  then resumes hitting. State does not bloat. `test_cross_process_resume` behavior
  does not regress.

### 5.4 Interaction with Feature A

Batching reduces the number of `model_turn` visits; caching removes repeated recall
on the visits that remain. The two reductions are multiplicative.

## 6. Error Handling and Edge Cases

| Case | Handling |
|---|---|
| Batch call schema-validation failure | Error tool_result for that call; batch not aborted; others run |
| Batch call denied_retry | Existing denied_retry message as that call's tool_result; others unaffected |
| Batch call malformed proposal | Reuse existing `_handle_malformed` per item |
| Approval mid-batch | Executed results committed; that call interrupts; `pending[i+1:]` deferred; re-issued next turn after resume |
| Recall cache empty after resume | First `model_turn` recomputes once, then hits |
| `propose_memory` not committed (approval/denied) | Cache **not** invalidated (store unchanged) |
| Run end | `RunRecallCache` run_id entry may persist (in-process, small). No LRU/TTL (YAGNI) |
| Concurrency within a run | None — graph advances a run serially; cache needs no lock |

### Explicitly Not Done (scope discipline)

- No true concurrent tool execution (deferred to a possible v0.6.f, decided by
  tool_latency metrics).
- No fixed plan→execute→synthesize orchestration.
- No structured-output hardening, preflight checks, or token/cache observability —
  these are other feedback items and separate future iterations.

## 7. Testing Strategy

TDD throughout, following existing layout (`tests/graph/`, `tests/memory/`,
`tests/graph/runtime/`).

**Batch tool execution** (`tests/graph/test_nodes.py`, possibly `test_loop.py`):

1. Model issues 3 tool calls in one turn → one `execute_tool_node` visit produces 3
   tool_results, `pending_tool_calls` cleared, **no deferred messages**.
2. Protocol invariant: N tool_use ↔ N tool_result (well-formed history).
3. Second call in batch fails schema → it gets an error, calls 1 and 3 still
   executed (error isolation).
4. Second call triggers interrupt → call 1's result kept, approval raised, call 3
   marked deferred; after resume, call 3 re-issued.
5. Side-effect order: two `save_draft` calls write in `pending` order.
6. Turn-count regression: reconstruct the log scenario (two `fetch_url` same turn) →
   assert `model_turn` invocation count drops sharply (from per-call turns to one).

**Recall cache** (`tests/memory/` or `tests/graph/test_nodes.py`, with a
spy/counting fake store):

7. Three consecutive `model_turn` in one run, no writes → store recall/select called
   **exactly once** (cache hit).
8. A `save_memory` commits mid-run → cache invalidated → next `model_turn`
   recomputes once (call count +1), and the new record appears in the selected set.
9. `propose_memory` reaching only `approval_required` → cache **not** invalidated
   (call count unchanged).
10. Cross-process resume (extend `test_cross_process_resume`): first `model_turn`
    after resume recomputes once, result equivalent to the uninterrupted path.

**Regression guard:** run full `uv run pytest -q` (currently 592 passed) to confirm
no regression; ruff/mypy clean on touched files.

## 8. Acceptance Criteria

- `execute_tool_node` processes all pending tool calls in one node visit, with one
  tool_result per call, deferring only the tail after a mid-batch approval interrupt.
- Per-call errors do not abort the batch; side effects run in `pending` order.
- Memory recall/selection is computed once per run and reused until a committed
  memory write invalidates it; cross-process resume recomputes once and stays
  correct.
- The recall cache is in-process on `GraphDeps`, not in checkpoint state.
- Full test suite green; new tests cover batch execution, error isolation, mid-batch
  approval, recall caching, invalidation-on-write, and resume.

## 9. Non-Goals

- Do not add concurrency to tool execution.
- Do not introduce fixed orchestration or planning into the runtime.
- Do not change the graph topology or `model_turn_node` context assembly beyond
  wrapping the recall pair in the cache.
- Do not store memory blocks in checkpoint state.
- Do not address the other feedback items (structured output, preflight, metrics) in
  this release.
