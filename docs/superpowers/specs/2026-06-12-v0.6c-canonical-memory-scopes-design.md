# V0.6.c Design Spec ‚Äî Canonical Memory Scopes

## 1. Purpose

V0.6.c removes the legacy Memory scope names introduced before the core concept
alignment. Memory now uses only first-principles scopes:

```text
user
workspace
agent
thread
```

The release is intentionally breaking. It does not keep `project` or
`conversation` as accepted Memory scopes, storage directories, tool schema
values, or policy branches.

## 2. Problem

V0.6.b clarified that `project` means `workspace` and `conversation` means
`thread`, but the runtime still stored records under:

```text
memory/project/<project_key>/
memory/conversation/<thread_id>/
```

That physical layout leaks old concepts into the user-visible file tree. It also
makes examples harder to explain: users see `workspace` in prompts and
`project` on disk, or `thread` in APIs and `conversation/session` on disk.

## 3. Non-Goals

- Do not read legacy `memory/project` or `memory/conversation` directories.
- Do not migrate old records automatically.
- Do not keep alias support in schemas or helper functions.
- Do not add a new Memory backend.
- Do not rename `MemoryType.project` in this release. That type describes a
  workspace-level category and can be renamed separately if it proves worth the
  churn.
- Do not change Workspace file layout except where examples point Memory to the
  project-local `.modi/memory` root.

## 5. Scope Semantics

- `user`: durable preferences and facts for the current user.
- `workspace`: durable rules, facts, and decisions for the current work
  boundary.
- `agent`: reusable methods and feedback for a specific agent definition.
- `thread`: short-lived continuity inside one task chain.

Lookup precedence becomes:

```text
thread -> workspace -> agent -> user
```

## 6. Runtime Changes

### Types

`MemoryScope` becomes:

```python
Literal["user", "workspace", "agent", "thread"]
```

`MemoryBlock.scope` and all public Memory APIs use the same set.

### Scope Keys

`MemoryScopeKeys` becomes:

```python
class MemoryScopeKeys:
    user_key: str
    workspace_key: str
    agent_name: str
    thread_id: str
```

There is no normalization function. `workspace` resolves to
`memory/workspace/<workspace_key>/`; `thread` resolves to
`memory/thread/<thread_id>/`.

### Store

`MemoryPaths` becomes:

```python
MemoryPaths(
    user=<memory_root>/user,
    workspace=<memory_root>/workspace,
    agent=<memory_root>/agent,
    thread=<memory_root>/thread,
)
```

The store does not search `project` or `conversation`.

### Session

`ModiSession` constructs canonical paths and scope keys from:

- `user_key`: local default unless future settings provide a value.
- `workspace_key`: readable key from the configured workspace run-file root
  when it has a specific name, such as `research_assistant`; generic roots such
  as `.modi/workspace` fall back to a stable work-boundary fingerprint.
- `agent_name`: current top-level agent or current run agent.
- `thread_id`: active thread id; direct session memory calls outside a run use
  `session`.

### Builtin Tools

- `recall_memory.scopes`: `user`, `workspace`, `agent`, `thread`.
- `save_memory.scope`: `thread`, `agent`.
- `propose_memory.scope`: `user`, `workspace`, `agent`, `thread`.

`project` and `conversation` are schema-invalid.

### Policy

- `thread` and `agent` writes may be allowed by default.
- `user` and `workspace` writes require approval by default.
- Unknown scopes are denied.

## 7. Documentation Changes

Update current architecture and implementation docs so they no longer describe
`project` or `conversation` as active Memory scopes. Historical release specs
remain historical and may mention old names.

Required current docs:

- `docs/architecture/12-memory-store.md`
- `docs/implement/14-memory-store.md`
- `docs/types-reference.md`
- `CHANGELOG.md`
- `docs/changelog.md`

## 8. Example Changes

`examples/research_assistant` must demonstrate the canonical layout:

```text
.modi/memory/user/default/
.modi/memory/workspace/research_assistant/
.modi/memory/agent/research-assistant/
.modi/memory/thread/<thread_id>/
```

The example must not create `.demo_memory`, `memory/project`, or
`memory/conversation`.

## 9. Tests

Tests must assert:

- workspace records write to the same readable workspace key used by run-file
  storage, for example `memory/workspace/research_assistant/`
- thread records write to `memory/thread/<thread_id>/`
- builtin schemas reject `project` and `conversation`
- policy does not special-case old scopes
- research assistant example does not create old directories
- full test suite passes

## 10. Acceptance Criteria

- `rg` over current source and current docs finds no active Memory alias helper
  such as `normalize_memory_scope`.
- `MemoryScope` contains only `user`, `workspace`, `agent`, and `thread`.
- `MemoryPaths` contains only `user`, `workspace`, `agent`, and `thread`.
- Running the research assistant example writes canonical memory directories.
- Full test suite passes.
- Changelog marks V0.6.c as a breaking Memory layout cleanup.
