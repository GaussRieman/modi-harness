# V0.6.d Design Spec — Model-First Harness

## 1. Purpose

V0.6.d corrects the architecture posture of Modi Harness.

Harness is not a second decision-maker beside the model. Harness is the model's
execution substrate: it gives the model context, tools, file access, memory
access, output validation, safety boundaries, and traceability.

The primary flow is:

```text
Human natural language
-> Model reasoning
-> Harness execution surface
-> tools, files, memory, trace, output
```

Not:

```text
Human
-> Harness decides
-> Model follows
```

## 2. First Principles

If a model had unlimited context, perfect tool use, reliable memory, safe
execution, and perfect persistence, Harness would be unnecessary.

Harness exists because real model execution needs support:

- context must be assembled into a usable window
- tools must be exposed through stable schemas
- files must be read and written safely
- memory must be persisted and retrieved
- risky side effects need boundaries
- outputs may need validation
- runs must be observable and resumable

Therefore, Harness should extend model capability rather than compete with model
reasoning.

## 3. Architectural Position

The model is primary.

Harness is:

- an execution surface
- a context assembler
- a tool router
- a file and memory substrate
- a policy boundary
- an observation layer

Harness is not:

- a planner that replaces the model
- a hidden product manager
- an autonomous memory governor
- a parallel agent deciding what the user meant
- a framework vocabulary users must learn before agents work

## 4. Design Principles

### 4.1 Natural-Language First

Users express intent in natural language. Harness should not require users to
understand Memory, Workspace, Trace, draft, artifact, scope, or policy concepts
before a task can run.

### 4.2 Model-Selected Actions

Models choose whether to use available tools. Harness should expose clear tool
affordances and execute the model's selected action within policy boundaries.

### 4.3 Context As Interface

The main Harness output before each model step is context. Context should give
the model enough relevant information to act without forcing all stored state
into the prompt.

### 4.4 Tools As Affordances

Builtin tools should describe what the model can do:

- recall memory
- read workspace files
- save drafts or artifacts
- propose memory
- call external tools

Tool descriptions carry usage guidance. Agent prompts should not repeat Harness
internals.

### 4.5 Policy As Boundary

Policy gates side effects. It should prevent unsafe or irreversible actions,
ask for approval when required, and record decisions. It should not become a
planner for ordinary model behavior.

### 4.6 Memory As Model-Accessible Long-Term Context

Memory is a model-accessible long-term context tool. The model may recall memory
or propose memory. Users and applications may also write memory explicitly.

Harness stores, retrieves, validates, and constrains memory operations. It
should not be framed as an independent agent that decides what matters.

### 4.7 Workspace As Model-Accessible Files

Workspace run files are the model/tool file boundary. Inputs, references,
drafts, artifacts, state snapshots, and logs appear only when useful.

Workspace should feel like the model's working desk, not a separate project
management abstraction.

### 4.8 Trace As Observation

Trace records what happened. It supports debugging, audit, replay, and eval.
Trace does not enter model context unless an application or user explicitly
turns it into task input or memory.

## 5. Agent Authoring Implications

Agent authors should write domain behavior, not Harness internals.

Good agent instruction:

```text
你是研究助手。调查用户问题，评估来源，并生成带证据的中文研究简报。

要求：
- 给出关键结论和对应证据。
- 区分确定结论和未决问题。
- 标注置信度和风险。
- 需要时使用可用工具查找资料、读取输入文件、保存结果或查询记忆。
```

Bad agent instruction:

```text
Memory is not trace/log. Drafts and artifacts are Workspace outputs. Use
user/workspace/thread/agent scopes...
```

The latter is framework training. It belongs in tool descriptions, runtime
policy, or developer docs, not in ordinary agent prompts.

## 6. Runtime Implications

### Context

Context assembly should be helpful but minimal:

- include current task
- include selected relevant memory when useful
- include input refs and workspace refs
- include tool descriptions
- include output contract

Context should not become a dump of all files, all memory, or trace.

### Memory

Automatic memory selection may preload a small number of relevant records, but
it is a model aid, not Harness taking over reasoning. `recall_memory` remains a
model-selected tool for additional search.

Model-facing writes should remain proposals. Durable writes still need policy
boundaries and, when appropriate, human approval.

### Workspace

Run files should be lazily materialized. A run with only a trace should not show
empty input, reference, state, draft, or artifact directories.

Caller-provided input files are created at run start and exposed to the model as
workspace refs. The model or tools decide whether to read them.

### Policy

Policy should answer:

```text
May this requested action execute?
```

It should not answer:

```text
What should the model do next?
```

### Trace

Trace should remain observation. It records context builds, model calls, tool
calls, policy decisions, memory operations, output validation, and errors.

## 7. Documentation Implications

Current docs should be rewritten over time to avoid overstating Harness as an
autonomous governing system.

Preferred wording:

- "Harness exposes tools to the model."
- "Harness assembles context for the model."
- "Harness enforces policy boundaries."
- "Harness records trace for observation."

Avoid:

- "Harness decides what the model should remember."
- "Harness manages project state."
- "Harness controls the agent."
- "Harness is the agent brain."

## 8. Example Implications

Examples should demonstrate natural model use:

- concise agent prompts
- clear output requirements
- available tools described by schemas
- minimal Harness vocabulary in the agent file

The research assistant example should eventually remove framework-heavy
instructions about Memory, Workspace, Trace, draft, artifact, and scope from the
agent prompt. The demo can still use those systems; it should not require the
agent author to explain them.

## 9. Non-Goals

- Do not remove policy, trace, memory, or workspace.
- Do not make model tool calls unconstrained.
- Do not hide tool descriptions from the model.
- Do not collapse all runtime behavior into a single prompt.
- Do not make this release a large runtime rewrite.

## 10. Acceptance Criteria

- Architecture docs state that the model is primary and Harness is execution
  substrate.
- Core concepts are reframed as model-visible or model-supporting surfaces.
- Agent authoring guidance says agent prompts should avoid Harness internals.
- Changelog records V0.6.d as a model-first architecture posture correction.
- Future implementation plans can reduce framework-heavy example prompts and
  move usage guidance into builtin tool descriptions and runtime behavior.
