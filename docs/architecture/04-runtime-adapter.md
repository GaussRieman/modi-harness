# Runtime Adapter

> **V0.2 status:** Runtime Adapter is now a thin wrapper around a compiled
> LangGraph graph. The hand-rolled state machine from V0.1 is gone. See
> [`16-subagent-runtime.md`](16-subagent-runtime.md) and
> [`17-checkpointer.md`](17-checkpointer.md) for the V0.2 additions that
> share this runtime.

Runtime Adapter maps Modi Harness onto a LangGraph compiled graph plus a
LangGraph `BaseCheckpointSaver`. Per `run_task` / `resume_task` / `approve_action` /
`reject_action`, it constructs the `RunnableConfig` (passing deps through
`config["configurable"]["modi_deps"]`), invokes the graph, flushes
`state["pending_trace_events"]` through `TraceMiddleware`, and translates the
final state into `RunTaskResponse`.

See [`types-reference.md`](../types-reference.md) for `AgentState`, `Message`,
`ToolCallRecord`, `DeniedAction`, `PendingApproval`.

## Position

Runtime Adapter is the only module that:

- builds the LangGraph compiled graph via `modi_harness.graph.build_main_graph`
- generates `run_id` and `root_run_id`, and seeds the initial state
- holds a reference to the checkpointer (from `build_checkpointer(settings)`)
- calls `graph.invoke(...)` / `graph.invoke(Command(resume=...))`
- flushes pending trace events through `TraceMiddleware`
- translates terminal state → `RunTaskResponse`

It does not implement policy, tool execution, model calls, output validation,
the step limit, or the repair budget inline. Those live in graph nodes
(`modi_harness.graph.nodes`) and the governance modules they call.

## Graph topology

```
START -> setup -> model_turn -> route_after_model -> execute_tool | validate_output
                      ^                                  |              |
                      |---- route_after_tool ------------+              |
                      |---- route_after_validate -----------------------+
```

- `setup` runs once: loads the agent profile, resolves skills, creates the
  workspace, seeds the run_start trace event.
- `model_turn` builds the ContextPack, calls the model, stages either
  `pending_tool_calls` or `pending_draft`.
- `execute_tool` calls `ToolGateway.execute_tool_call(...)` and may raise
  `langgraph.types.interrupt(...)` for approvals; the same node resumes when
  `Command(resume=...)` is delivered.
- `validate_output` runs `OutputController.validate(...)` and sets
  `final_output` / `status` accordingly.

## Identifiers

- `run_id` is generated by Runtime Adapter at run start (ULID).
- `root_run_id == run_id` for top-level runs; subagent dispatches propagate
  the parent's `root_run_id` (see `16-subagent-runtime.md`).
- `thread_id` is the LangGraph persistence key. Caller-supplied or auto-
  generated as `f"run_{ULID}"`. Subagent children get
  `f"{parent_thread}/sub/{ULID}"`.

## Loop

```text
load agent
-> load skills
-> build context
-> dispatch pre_model_call hooks
-> model step
-> dispatch post_model_call hooks
-> route response (tool or output)
-> tool gateway OR output controller
-> persist state
-> continue / interrupt / finish
```

Hook dispatch points: `session_start`, `user_prompt_submit`, `pre_model_call`, `post_model_call`, `pre_tool_use`, `post_tool_use`, `on_approval_request`, `on_denial`, `on_output_validate`, `on_run_end`, `on_error`. See [`13-hook-system.md`](./13-hook-system.md).

## Graph Nodes

V0.1 LangGraph nodes:

- `load_agent`
- `load_skills`
- `build_context`
- `model_step`
- `route_response`
- `execute_tool`
- `validate_output`
- `persist_state`

Conditional edges route to continue, interrupt, review, failure, or finish.

## Concurrency

- Each run owns its own workspace subdirectory; cross-run writes are forbidden.
- A workspace lock file guards state writes within a run; concurrent runs do not share locks.
- Two runs may execute in parallel against the same agent and project. Memory writes use Memory Store's per-scope lock.

## Rules

- V0.1 is single-agent only.
- LangGraph is the default runtime engine.
- State is persisted after every transition that changes behavior.
- Denied actions are not retried unchanged. The denied-retry guard runs both here (pre-dispatch) and in Tool Gateway (defense in depth).
- Hook feedback is stored as user feedback in `messages`.
- A blocking hook on `pre_tool_use` is equivalent to a user-initiated deny: tool does not execute and `DeniedAction` is recorded.
- Repair loops live here: malformed tool calls, schema-failed outputs, and validation issues trigger another model step rather than failing the run, up to a per-loop cap.
- Custom state machines are a fallback only when LangGraph blocks a required harness behavior.

## Boundaries

- Tool routing: Runtime Adapter selects the next node; Tool Gateway executes.
- Hook execution: dispatched here, performed by Hook System.
- State persistence: handled here, stored by Workspace Manager.
- Trace writes: emitted here as side effects of transitions; format owned by Trace Recorder.
