# Runtime Adapter

Runtime Adapter maps Modi Harness into a LangGraph execution loop.

See [`types-reference.md`](../types-reference.md) for `AgentState`, `Message`, `ToolCallRecord`, `DeniedAction`, `PendingApproval`.

## Position

Runtime Adapter is the only module that:

- builds the LangGraph state graph
- generates `run_id` and `root_run_id`
- enforces step limit
- decides continue / interrupt / finish
- writes state snapshots through Workspace Manager
- dispatches hook events

It does not implement policy, tool execution, model calls, or output validation inline.

## Identifiers

- `run_id` is generated by Runtime Adapter at run start.
- `root_run_id == run_id` for top-level runs; child runs (future subagent) carry the original.
- `thread_id` carries through across multiple `run_task` calls in a conversation.

## Loop

```text
load agent
-> load skills
-> build context
-> dispatch pre_model_call hooks
-> model step
-> dispatch post_model_call hooks
-> route response (tool or output)
-> tool gateway OR output controller
-> persist state
-> continue / interrupt / finish
```

Hook dispatch points: `session_start`, `user_prompt_submit`, `pre_model_call`, `post_model_call`, `pre_tool_use`, `post_tool_use`, `on_approval_request`, `on_denial`, `on_output_validate`, `on_run_end`, `on_error`. See [`13-hook-system.md`](./13-hook-system.md).

## Graph Nodes

V0.1 LangGraph nodes:

- `load_agent`
- `load_skills`
- `build_context`
- `model_step`
- `route_response`
- `execute_tool`
- `validate_output`
- `persist_state`

Conditional edges route to continue, interrupt, review, failure, or finish.

## Concurrency

- Each run owns its own workspace subdirectory; cross-run writes are forbidden.
- A workspace lock file guards state writes within a run; concurrent runs do not share locks.
- Two runs may execute in parallel against the same agent and project. Memory writes use Memory Store's per-scope lock.

## Rules

- V0.1 is single-agent only.
- LangGraph is the default runtime engine.
- State is persisted after every transition that changes behavior.
- Denied actions are not retried unchanged. The denied-retry guard runs both here (pre-dispatch) and in Tool Gateway (defense in depth).
- Hook feedback is stored as user feedback in `messages`.
- A blocking hook on `pre_tool_use` is equivalent to a user-initiated deny: tool does not execute and `DeniedAction` is recorded.
- Repair loops live here: malformed tool calls, schema-failed outputs, and validation issues trigger another model step rather than failing the run, up to a per-loop cap.
- Custom state machines are a fallback only when LangGraph blocks a required harness behavior.

## Boundaries

- Tool routing: Runtime Adapter selects the next node; Tool Gateway executes.
- Hook execution: dispatched here, performed by Hook System.
- State persistence: handled here, stored by Workspace Manager.
- Trace writes: emitted here as side effects of transitions; format owned by Trace Recorder.
